# Multi-Agent Review Checklists

Phase 4 各评审角色的详细检查清单。每个 Agent 启动时加载对应章节。

---

## Agent 1: Architect (架构师)

关注：结构合理性

```
- [ ] 分层是否清晰，模块边界是否合理
- [ ] 是否存在循环依赖或不合理的跨层调用
- [ ] 扩展性：未来需求变化时改动范围是否可控
- [ ] 一致性：命名、风格、模式是否与现有系统一致
- [ ] 复杂度：是否过度设计？有没有更简单的方案？
- [ ] 接口粒度：是否过粗（God API）或过细（chatty API）
- [ ] 领域边界：聚合根划分是否合理？跨聚合引用是否通过 ID？
```

---

## Agent 2: Security Expert (安全专家)

关注：安全漏洞

```
- [ ] 每个新接口是否有鉴权方案
- [ ] 敏感数据（手机号、身份证、银行卡）是否有脱敏/加密设计
- [ ] SQL 注入风险：是否使用参数化查询（#{} not ${}）
- [ ] 越权风险：是否有资源归属校验（用户只能访问自己的数据）
- [ ] 日志是否可能泄露敏感信息
- [ ] CSRF/XSS 防护（如有 Web 接口）
- [ ] 密钥/token 管理是否安全（不落库明文、不写日志）
- [ ] 文件上传/下载是否有路径遍历风险
```

---

## Agent 3: Performance Engineer (性能工程师)

关注：性能与容量

```
- [ ] 热路径识别：高频调用链是否有性能优化
- [ ] 数据库：查询是否走索引？是否有 N+1 问题？大表操作是否分批？
- [ ] 缓存设计：是否合理？穿透/击穿/雪崩是否有防护？
- [ ] 并发：是否有锁竞争？锁粒度是否合理？
- [ ] 容量预估：QPS / 数据量 / 连接数是否有估算？
- [ ] 异步化：可以异步的操作是否设计为异步？
- [ ] 批量操作：是否有不合理的循环单条处理？
- [ ] 连接池/线程池：是否有资源泄漏风险？
```

---

## Agent 4: Business Analyst (业务分析师)

关注：业务完整性

```
- [ ] 需求覆盖度：每个功能点是否都有对应设计
- [ ] 边界场景：并发操作、空数据、超大数据、重复提交
- [ ] 状态机：状态流转是否完整？是否有不可达/死锁状态？
- [ ] 业务规则：金额精度、时间窗口、限额规则是否明确
- [ ] 异常流程：失败重试、补偿机制、人工介入入口
- [ ] 用户体验：报错信息是否友好？操作是否可撤销？
- [ ] 数据一致性：最终一致还是强一致？用户能感知不一致吗？
```

---

## Agent 5: Codebase Archaeologist (代码考古学家)

关注：与现有系统的兼容性

```
- [ ] 现有接口契约：新设计是否破坏已有调用方
- [ ] 数据兼容：表结构变更是否需要数据迁移？存量数据影响？
- [ ] 依赖变更：新增/升级的依赖是否与现有版本兼容
- [ ] 配置变更：是否需要新增配置项？各环境是否都覆盖？
- [ ] 发布顺序：多服务变更是否有依赖顺序？能否无损发布？
- [ ] 回滚方案：能否快速回滚？回滚后数据是否一致？
- [ ] 代码复用：现有代码是否已有类似实现可复用？
- [ ] 技术债：是否引入新的技术债？是否有偿还计划？
```

---

## Agent 输出格式

所有 Agent 统一输出：

```json
[
  {
    "severity": "CRITICAL | HIGH | MEDIUM | LOW",
    "category": "架构 | 安全 | 性能 | 业务 | 兼容",
    "location": "文档章节编号，如 §3.1",
    "issue": "问题描述",
    "suggestion": "具体修改建议"
  }
]
```

## 评审报告汇总格式

```markdown
## 系分评审报告

### 统计
- CRITICAL: N 项（必须修改）
- HIGH: N 项（强烈建议修改）
- MEDIUM: N 项（建议改进）
- LOW: N 项（可选优化）

### CRITICAL Issues
| # | 视角 | 位置 | 问题 | 建议 |
|---|------|------|------|------|

### HIGH Issues
[同上格式]

### MEDIUM Issues
[同上格式]

### LOW Issues
[同上格式]
```
