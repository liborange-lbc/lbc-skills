# Review Checklists

本文件包含两个阶段的评审 checklist：
- **Part A**: Phase 4 设计评审（2 专家）
- **Part B**: Phase 8 代码与测试评审（3 专家）

---

# Part A: 设计评审 Checklists（Phase 4）

## Agent 1: 需求完整度专家

关注：功能拆解清单（requirement-breakdown.md）的每个子项是否在设计文档中有完整覆盖。

### 评审方法

```
对 requirement-breakdown.md 中的每一个 FR-XXX 子项，执行以下检查：

1. 【定位】在 design-doc.md 中搜索该 FR 编号或对应的功能描述
2. 【覆盖判定】判断设计内容是否充分：
   - ✅ 已覆盖：有完整的实现方案（接口设计 + 数据模型 + 流程说明）
   - ⚠️ 覆盖不足：有提及但缺少关键设计细节（如只有接口没有异常处理）
   - ❌ 未覆盖：设计文档中完全没有对应内容
3. 【严重度标记】
   - ❌ 未覆盖 → CRITICAL（功能缺失）
   - ⚠️ 覆盖不足 → HIGH（设计不完整）
```

### Checklist

```
- [ ] 每个 FR 子项在设计文档中有对应章节或段落
- [ ] 正常流程（happy path）的实现方案完整
- [ ] 异常/边界场景有明确的处理设计
- [ ] 权限控制要求在鉴权方案中有体现
- [ ] 数据变更在数据模型章节中有体现
- [ ] 功能间的依赖关系在流程图/时序图中有体现
- [ ] 所有 P0 优先级子项均已覆盖
```

### 输出格式

```markdown
## 需求完整度评审结果

### 覆盖度统计
- 功能子项总数: N
- ✅ 已覆盖: N
- ⚠️ 覆盖不足: N
- ❌ 未覆盖: N
- 覆盖率: xx%

### 逐项评审

| FR 编号 | 功能描述 | 覆盖状态 | 设计位置 | 问题说明 |
|---------|----------|----------|----------|----------|
| FR-A-001 | ... | ✅ | §3.1 | — |
| FR-A-002 | ... | ❌ | — | 设计文档中无对应实现方案 |
| FR-A-003 | ... | ⚠️ | §3.2 | 只有正常流程，缺少异常处理设计 |
```

## Agent 2: 架构安全专家

关注：架构合理性、安全性、与现有代码的兼容性和容错能力。

### Checklist — 架构合理性

```
- [ ] 分层是否清晰，模块边界是否合理
- [ ] 是否存在循环依赖或不合理的跨层调用
- [ ] 扩展性：未来需求变化时改动范围是否可控
- [ ] 一致性：命名、风格、模式是否与现有系统一致
- [ ] 复杂度：是否过度设计？有没有更简单的方案？
- [ ] 接口粒度：是否过粗（God API）或过细（chatty API）
```

### Checklist — 安全性

```
- [ ] 每个新接口是否有鉴权方案
- [ ] 敏感数据（手机号、身份证、银行卡）是否有脱敏/加密设计
- [ ] SQL 注入风险：是否使用参数化查询
- [ ] 越权风险：是否有资源归属校验
- [ ] 日志是否可能泄露敏感信息
- [ ] 密钥/token 管理是否安全
```

### Checklist — 兼容性与容错

```
- [ ] 新设计是否破坏已有接口的调用方
- [ ] 表结构变更是否需要数据迁移？存量数据影响？
- [ ] 新增/升级的依赖是否与现有版本兼容
- [ ] 发布顺序：多服务变更是否有依赖顺序？
- [ ] 回滚方案：能否快速回滚？回滚后数据是否一致？
- [ ] 降级策略：新逻辑故障时是否影响现有功能？
- [ ] 配置变更：是否需要新增配置项？各环境是否覆盖？
```

# Part B: 代码与测试评审 Checklists（Phase 8）

## Agent 1: 代码规范专家

关注：编码标准、安全编码实践。

### 通用 Checklist（所有语言）

```
- [ ] 无硬编码的密钥、密码、token、连接字符串
- [ ] 无 TODO/FIXME/HACK 未处理的标记
- [ ] 无注释掉的代码块
- [ ] 函数/方法不超过 50 行
- [ ] 文件不超过 800 行
- [ ] 圈复杂度不超过 10（嵌套不超过 4 层）
- [ ] 命名清晰、自解释
- [ ] 错误处理完整，不吞异常
- [ ] 资源正确释放（连接、流、锁）
```

### Java 专项

```
- [ ] 类名 PascalCase、方法 camelCase、常量 UPPER_SNAKE_CASE
- [ ] 异常处理：不用 catch(Exception e) 一把抓、自定义异常有继承体系
- [ ] 日志：使用参数化（{}）、不拼接字符串、不打印敏感信息
- [ ] @Transactional：范围最小化、只在 Service 层使用、注意传播行为
- [ ] Spring Bean：优先构造器注入、避免 @Autowired 字段注入
- [ ] MyBatis：使用 #{} 参数化（非 ${}）、注意 N+1 查询、批量操作
- [ ] 线程安全：共享状态加锁或使用不可变对象、ThreadLocal 使用后清理
- [ ] 集合操作：优先 Stream API、注意 null 安全
```

### Python 专项

```
- [ ] 遵循 PEP 8 风格
- [ ] 函数签名和返回值有类型注解
- [ ] 异常处理：捕获具体异常类型、使用 contextmanager
- [ ] 异步代码：async/await 正确使用
- [ ] 导入顺序：标准库 → 第三方 → 本地
```

### JavaScript/TypeScript 专项

```
- [ ] TypeScript strict 模式下无 any 类型
- [ ] async/await 统一使用，无混用 .then()
- [ ] 无未处理的 Promise rejection
- [ ] 模块导入无循环依赖
- [ ] React（如适用）：Hooks 规则遵循、组件职责单一
```

## Agent 2: 逻辑正确性专家

关注：代码实现与设计文档的一致性、逻辑正确性。

### Checklist

```
- [ ] 每个 FR 子项的实现与 design-doc.md 中的方案一致
- [ ] 接口签名与设计文档中的 API 定义一致（路径、参数、返回值）
- [ ] 数据模型与设计文档中的表结构变更一致
- [ ] 业务规则实现与需求描述一致（金额精度、时间窗口、限额等）
- [ ] 空值/null 检查：所有外部输入在使用前都有校验
- [ ] 边界条件：数值范围、字符串长度、集合大小
- [ ] 并发安全：共享资源的竞争条件、幂等实现
- [ ] 事务边界：@Transactional 范围与设计一致
- [ ] 异常路径：每个可能的异常都有对应的处理逻辑
- [ ] 资源管理：数据库连接、HTTP 连接、文件句柄正确释放
```

## Agent 3: 测试质量专家

关注：测试用例是否**真正验证**了系统功能。

### Checklist

```
- [ ] 每个 FR 子项至少有一个测试用例
- [ ] 测试有有意义的断言（不只是 assertNotNull / assertTrue(true)）
- [ ] 断言验证的是**业务行为**而非**实现细节**
- [ ] 测试数据贴近真实业务场景（非 "test123", "aaa"）
- [ ] 异常路径有测试（输入校验失败、依赖超时、并发冲突）
- [ ] 测试之间相互独立，无顺序依赖
- [ ] Mock 行为与真实依赖一致（返回值、异常类型）
- [ ] 未过度 Mock（能用内存数据库就别 Mock 数据层）
- [ ] 测试命名清晰，能从名字看出测试什么场景
- [ ] 测试能稳定通过，无随机失败（flaky test）
```

### 测试有效性审计方法

对每个测试用例执行：

1. **读测试代码** — 理解测试在做什么
2. **读被测代码** — 确认测试覆盖了关键路径
3. **检查断言** — 断言是否验证了声称要测试的功能
4. **判定有效性**：
   - ✅ 有效：断言与功能描述一致，覆盖关键路径
   - ⚠️ 待改进：有断言但不够充分（缺少边界检查、断言过于宽泛）
   - ❌ 无效：断言与功能无关、或无实质断言

# 通用输出格式

所有评审 Agent 统一输出：

```json
[
  {
    "severity": "CRITICAL | HIGH | MEDIUM | LOW",
    "category": "需求覆盖 | 架构 | 安全 | 兼容 | 规范 | 逻辑 | 测试",
    "location": "文档章节/文件路径:行号",
    "issue": "问题描述",
    "suggestion": "具体修改建议"
  }
]
```

```markdown
### MEDIUM Issues

### LOW Issues
[同上格式]
```
