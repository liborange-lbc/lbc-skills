# Requirement Interview & Breakdown Checklist

Phase 1 的详细面试清单和功能拆解规范。

---

## Part A: 需求面试（Phase 1a）

逐项确认，全部明确后才进入功能拆解。

### 必问项

| # | 维度 | 问题 | 目的 |
|---|------|------|------|
| 1 | 业务背景 | 为什么做？解决什么业务问题？ | 避免设计偏离业务目标 |
| 2 | 核心功能 | 具体做什么？逐条列出功能点 | 明确设计范围 |
| 3 | 影响范围 | 涉及哪些模块/服务/团队？ | 识别协作依赖 |
| 4 | 用户角色 | 谁用？几种角色？权限差异？ | 鉴权和数据隔离设计依据 |
| 5 | 数据流向 | 数据从哪来、到哪去、中间怎么变？ | 接口和存储设计依据 |
| 6 | 非功能需求 | QPS 预估？延迟要求？数据量级？ | 性能和容量设计依据 |
| 7 | 约束条件 | deadline？兼容性要求？不能动的部分？ | 划定设计边界 |
| 8 | 验收标准 | 怎么算做完了？ | 设计完成后的自检依据 |

### 追问策略

- 用户回答模糊时：给出 2-3 个具体选项让用户选择，而非开放式追问
- 用户说"你看着办"时：给出你的假设，请用户确认或否定
- 涉及多团队时：确认谁是接口人、谁做决策

### 产出格式

```markdown
## 需求确认清单

| 维度 | 确认内容 | 状态 |
|------|----------|------|
| 业务背景 | [具体描述] | [已确认/待确认] |
| ... | ... | ... |

确认人：[用户]
确认时间：[日期]
```

## Part B: 功能拆解（Phase 1b）

在需求确认后，将需求拆解为**不可再分**的功能子项。

### 拆解维度

对每个核心功能，从以下维度展开：

| 维度 | 说明 | 示例 |
|------|------|------|
| **正常流程** | 主路径上的每个步骤 | 创建订单、扣减库存、生成支付单 |
| **异常处理** | 每个步骤可能的失败场景 | 库存不足、支付超时、重复提交 |
| **权限控制** | 谁能执行、谁不能执行 | 仅商户管理员可操作、普通用户只读 |
| **数据变更** | 创建/修改/删除了哪些数据 | 创建订单记录、更新库存数量 |
| **边界条件** | 极端输入和特殊状态 | 空列表、超大数据量、并发操作 |
| **状态流转** | 数据状态的变化路径 | 待支付→已支付→已发货→已完成 |

### 拆解粒度标准

一个功能子项是否粒度合适，用以下标准判断：

- ✅ 可以用一句话描述清楚
- ✅ 可以用一个测试用例验证
- ✅ 实现时对应代码中一个明确的逻辑分支或操作
- ❌ 如果需要"并且"连接两个动作，应该拆分

### 编号规则

```
FR-{模块缩写}-{三位序号}

示例：
FR-ORDER-001   订单模块第 1 个子项
FR-PAY-003     支付模块第 3 个子项
FR-INV-010     库存模块第 10 个子项
```

### 优先级定义

| 优先级 | 定义 | 处理方式 |
|--------|------|----------|
| P0 | 核心功能，无此功能需求不成立 | 必须设计、必须实现、必须测试 |
| P1 | 重要功能，影响用户体验或数据完整性 | 必须设计、应该实现 |
| P2 | 增强功能，可延后实现 | 记录在案，本期可不实现 |

### 产出格式

```markdown
# 功能拆解清单

> 需求: {feature-name}
> 拆解时间: {date}
> 子项总数: N（P0: x, P1: y, P2: z）

## 模块 A: {模块名}

| 编号 | 功能子项 | 类型 | 优先级 | 备注 |
|------|----------|------|--------|------|
| FR-A-001 | {一句话描述} | 正常流程 | P0 | |
| FR-A-002 | {一句话描述} | 异常处理 | P0 | 依赖 FR-A-001 |
| FR-A-003 | {一句话描述} | 权限控制 | P1 | |
| FR-A-004 | {一句话描述} | 边界条件 | P1 | 并发场景 |

## 模块 B: {模块名}

| 编号 | 功能子项 | 类型 | 优先级 | 备注 |
|------|----------|------|--------|------|
| FR-B-001 | ... | ... | ... | |

---

> 用户确认：[待确认/已确认]
> 确认时间：[日期]
```

### 自检清单

完成拆解后，用以下清单自检：

```
- [ ] 每个核心功能的正常流程都有子项
- [ ] 每个可能失败的步骤都有异常处理子项
- [ ] 涉及权限的操作都有权限控制子项
- [ ] 涉及数据变更的操作都明确了变更内容
- [ ] 状态流转完整，无不可达或死锁状态
- [ ] 编号连续无遗漏
- [ ] 每个子项都满足粒度标准（一句话可描述、一个测试可验证）
- [ ] P0 子项已覆盖所有核心路径
```
