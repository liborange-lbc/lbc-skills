## 关键发现

### 1. 数据结构完整可解析

所有 log 文件均为 Markdown 格式，结构化程度高：
- 评审报告首行为判定行 `## 判定：PASS/FAIL`，可用正则提取
- Agent注册表为标准 Markdown 表格，可逐行解析
- 时间线为 `- YYMMDD HHmm {事件}` 格式，可用正则提取
- 迭代日志文件名含编号 `phase{N}-迭代-{M}.md`，可用 glob 扫描

### 2. P4 角色数量存在文档不一致

- SKILL.md L88 说"4个评审"，L162-170 角色表只列3个
- 执行日志规范的 Agent 注册表示例列了4个（含 security-reviewer）
- 产出目录结构列了 `phase4-评审-安全.md`
- 系分评审执行指南只定义3个角色

**结论**: Dashboard 应动态扫描文件，不硬编码角色数量。

### 3. 两种显示模式

- 无数据模式: 展示 8 Phase 设计理念（R11 需求）
- 有数据模式: 展示实时运行状态

前端需根据后端返回数据是否为空来切换。

### 4. 需求状态存储在需求清单.md中

需求状态（⏳/📐/💻/✅/⚠️）作为 Markdown 表格的一列存储在 `需求清单.md` 中，后端需解析此表格提取状态。
