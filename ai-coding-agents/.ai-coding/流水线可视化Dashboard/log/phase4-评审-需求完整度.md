## 问题清单

### [MEDIUM] R02 设计理念融入方式与验收标准表述不一致
- 需求ID: R02
- 描述: 需求验收标准写"每个阶段节点展开后可看到该阶段体现的设计理念说明"，暗示需要一个"展开"交互才能显示设计理念。系分 §10 的实现方案是将设计理念融入各 UI 元素（tooltip 悬浮、连线标注等），并非"节点展开"。两者行为不同：一个需要点击展开，一个需要鼠标悬浮。
- 建议: 在系分或需求清单中明确统一交互形式。建议在系分 §10 补充说明："节点 tooltip 悬浮即等同验收标准中的'展开后可见'"，或更新需求验收标准为"节点悬浮时可见设计理念"。

### [MEDIUM] R05 "进行中动画"验收标准无对应设计
- 需求ID: R05
- 描述: R05 验收标准明确要求"进行中的节点有视觉区分（如高亮或动画）"。系分 §5.3 为 `running` 状态定义了浅蓝色背景+蓝色边框，但未设计任何动画效果（如 CSS pulse 动画或闪烁边框）。颜色区分能够满足"视觉区分"的基本要求，但与验收标准中"动画"的可选项存在差距。
- 建议: 在系分 §5.3 或 §4.2 中明确说明：`running` 状态仅使用颜色区分，不添加动画（保持简洁原则），同时更新 R05 验收标准去掉"或动画"，或在 style.css 组件中补充一个轻量 CSS animation（如 2s 边框闪烁）。

### [MEDIUM] R06 "按需求隔离"设计实现不完整
- 需求ID: R06（关联 §10 原则 6）
- 描述: 需求 R06 的核心是"并行 Agent 信息同时展示"，已有 §6.2 覆盖。但需求清单中 R06 验收标准与设计理念"按需求隔离"共用同一编号（需求清单 §10 表格第 6 行"按需求隔离"对应顶部需求选择器），而 §3.1 API /api/status 的 Response 结构仅返回单一需求目录数据，未说明多需求切换时后端如何动态切换扫描目录（选择器选择后如何传参给后端）。
- 建议: 在 §3.1 GET /api/status 中补充可选查询参数 `?dir=` 说明，或在 §3.3 启动方式中说明需求选择器是通过切换前端 URL 参数还是通过后端重新加载实现目录切换。

### [LOW] R12 追踪矩阵引用章节不完整
- 需求ID: R12
- 描述: 追踪矩阵中 R12 引用了 §4.1 页面布局和 §8.3 面板样式，但未引用 §5.3 状态颜色映射（浅色状态色块也是视觉风格的一部分）。
- 建议: 在追踪矩阵 R12 行的"设计章节"列补充 `§5.3 状态颜色映射`。

### [LOW] NFR03 追踪矩阵未说明指标收紧关系
- 需求ID: NFR03
- 描述: 验收标准为响应时间 < 500ms，系分 §11.3 设计目标为 < 100ms（比验收标准严 5 倍）。虽然更严格的设计完全能满足验收标准，但追踪矩阵未注明两者的对应关系，可能给后续开发者造成困惑（误以为是两个不同指标）。
- 建议: 在追踪矩阵 NFR03 的"实现方案"列补注："文件扫描目标 < 100ms（优于验收标准 500ms）"。
