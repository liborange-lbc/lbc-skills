# 知识摘要

## 对 Dashboard 开发直接有用的知识

### 核心流程

8 Phase 流水线顺序执行，P4/P6 含并行分支和修正循环：

```
P1 → P2 → P3 → P4(3并行评审) ⇄ 修正(≤3轮) → P5 → P6(3并行评审) ⇄ 修正(≤3轮) → P7 → P8
```

- P4/P6 FAIL 时触发修正循环：Resume原Agent修复 → 重新评审
- 最多 MAX_ITERATION（默认3）轮
- 超限后标记 ⚠️ 列入残留风险

### 判断 Phase 状态的规则

| 状态 | 判断依据 |
|------|---------|
| 未开始 | 对应 log 文件不存在 |
| 进行中 | log 文件存在但无判定行/无"完成"标记 |
| 已完成(PASS) | 评审报告含 `## 判定：PASS`；或其他 log 文件中有"完成"标记 |
| 失败(FAIL) | 评审报告含 `## 判定：FAIL` |
| 修正中 | 存在 `phase{N}-迭代-{M}.md` 且最新迭代无 PASS |
| 低质量通过 | 需求状态为 ⚠️ |

### 从执行日志提取 Agent 注册信息

解析 `执行日志.md` 中 `## Agent 注册表` 下的 Markdown 表格：
- 每行一个 Agent: 角色 | agentId | Phase | 状态 | 备注
- 状态值: ⏳ 待启动 / 🔄 修正中 / ✅ 完成 / ❌ 已废弃 / ⚠️

### 从执行日志提取时间线

解析以 `- YYMMDD HHmm` 开头的行：
- 格式: `- {时间} {事件描述}`
- 事件关键词: "启动"/"完成"/"结果"/"修正"/"PASS"/"FAIL"

## 状态模型

### 需求状态生命周期 [来源: SKILL.md:262-270]

| 状态符号 | 含义 | 触发阶段 |
|----------|------|---------|
| ⏳ | 待处理 | P1 初始 |
| 📐 | 已设计 | P3 完成 |
| 💻 | 已编码 | P5 完成 |
| ✅ | 已验证 | P7 通过 |
| ⚠️ | 低质量通过 | 3轮修正后仍有问题 |

**转换规则**: ⏳ → 📐（designer完成设计）→ 💻（coder完成编码）→ ✅（tester验证通过）。如修正循环3轮后仍有问题 → ⚠️。

### Phase 状态

| 状态 | 说明 |
|------|------|
| pending | 未开始 |
| running | 执行中 |
| pass | 完成/通过 |
| fail | 评审失败 |
| iterating | 修正循环中 |

### 评审判定 [来源: reference/评审检查清单.md:122-128]

| 条件 | 结果 |
|------|------|
| 0 CRITICAL + 0 HIGH + 需求覆盖/实现100% | **PASS** |
| 存在 CRITICAL/HIGH 或需求未覆盖 | **FAIL** → 修正循环 |

### 问题严重度 [来源: reference/评审检查清单.md:121]

CRITICAL > HIGH > MEDIUM > LOW

## 8 大设计原则与界面体现建议 [来源: SKILL.md:336-345]

| # | 设计原则 | 含义 | 建议在哪个节点体现 |
|---|----------|------|------------------|
| 1 | 主Agent纯调度 | 不执行、只决策和路由 | P8 节点说明（主Agent唯一直接产出） |
| 2 | 需求驱动全流程 | 需求清单贯穿8个Phase | P1节点 + 每个Phase的输入中标注"需求清单" |
| 3 | 文档驱动交接 | Phase间通过文档传递 | 连线上标注传递的文档名 |
| 4 | Resume优于新建 | 修正循环用SendMessage复用Agent | P4/P6 修正循环说明 |
| 5 | 经验库累积 | 修正中沉淀的模式跨需求复用 | P4/P6 修正后标注"→经验库" |
| 6 | 按需求隔离 | 不同需求独立目录 | 顶部需求选择器 |
| 7 | GitNexus赋能 | 代码分析用知识图谱 | P2/P7 节点说明 |
| 8 | 扩展优于硬编码 | EXTENSION_DOCS挂载外部知识 | 各Phase输入中标注"扩展文档(可选)" |

## 修正循环关键信息 [来源: reference/修正循环协议.md]

### 问题分类与回退深度

| 问题性质 | 回退到 |
|----------|--------|
| 局部问题 | Resume 原Agent 定点修复 |
| 架构级问题 | 回退上一阶段重做 |
| 需求理解错误 | 回退 Phase 1 全流程重来 |

### 迭代文件命名

- P4 修正: `log/phase4-迭代-1.md`, `log/phase4-迭代-2.md`, ...
- P6 修正: `log/phase6-迭代-1.md`, `log/phase6-迭代-2.md`, ...
- 最多 MAX_ITERATION（默认3）轮

### 经验库沉淀时机

每次修正循环完成后，将本轮修复揭示的可复用模式追加到 `经验库.md`。
