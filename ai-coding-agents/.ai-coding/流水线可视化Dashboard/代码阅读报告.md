## 关键数据结构

### 1. 执行日志格式 [来源: reference/执行日志规范.md]

**时间格式**: `YYMMDD HHmm`，如 `260501 0930`

**主Agent执行日志** (`log/执行日志.md`) 包含三个核心区域：

1. **项目信息区**: 启动时间、需求来源、配置参数（BATCH_SIZE, MAX_ITERATION）、GitNexus索引名
2. **Agent注册表**: 表格形式记录所有子Agent的 agentId、所属Phase、状态、备注
3. **时间线**: 按时间顺序记录每个Phase的开始/结束、修正循环、批次分隔等事件

**Agent注册表格式** [来源: reference/执行日志规范.md:24-37]:

```
| 角色 | agentId | Phase | 状态 | 备注 |
|------|---------|-------|------|------|
| planner | a1b2c3d4 | 2 | ✅ 完成 | |
| designer | e5f6g7h8 | 3 | 🔄 修正中 | 第1轮修正 |
```

状态值: `⏳ 待启动` | `🔄 修正中` | `✅ 完成` | `❌ 已废弃` | `⚠️`

**时间线格式** [来源: reference/执行日志规范.md:47-73]:
```
- 260501 0930 项目启动
- 260501 0957 Phase 4 启动 4 个评审:
    - design-reviewer → agentId: i9j0k1l2
- 260501 1002 Phase 4 结果: C=0 H=1 M=2 L=1 覆盖=100% → FAIL
- 260501 1003 修正第1轮: SendMessage(to: e5f6g7h8) Resume designer
- 260501 1007 ── Batch 1: R01-R03 ──
```

统计缩写: C=CRITICAL H=HIGH M=MEDIUM L=LOW

### 2. 评审报告格式 [来源: subagent/phase04-系分评审员/系分评审执行指南.md:56-78]

所有评审报告（P4系分评审 + P6代码CR）共享统一格式：

```markdown
# {系分评审报告|代码CR报告} — {角色名}

## 判定：PASS / FAIL

## 统计
- CRITICAL: N 项
- HIGH: N 项
- MEDIUM: N 项
- LOW: N 项

## 需求覆盖度（仅 req-completeness-reviewer / code-req-reviewer）
| 需求ID | 描述 | ... | 判定 |

## 问题清单
| # | 严重度 | 位置 | 问题 | 修改建议 | 关联需求 |
```

**关键**: 报告第一行必须是 `## 判定：PASS` 或 `## 判定：FAIL`，主Agent用 Grep 提取。

### 3. 迭代日志格式 [来源: reference/修正循环协议.md:99-113]

## 各 Phase 详解

### Phase 1: 需求澄清

- **执行者**: interviewer 子Agent
- **输入**: 用户原始需求 + EXTENSION_DOCS.requirement（可选）
- **输出**: `{需求目录}/需求清单.md` + `log/phase1-需求澄清.md`
- **职责**: 8维度面试（业务背景/核心功能/影响范围/用户角色/数据流向/非功能需求/约束条件/验收标准），拆解为原子级需求条目（R01-RNN + NFR01-NFRNN），每条需求可独立测试和验收
- **编号规则**: 功能需求 R{NN}、非功能需求 NFR{NN}、优先级 P0>P1>P2
- **设计理念体现**: 需求驱动全流程 — 需求清单贯穿8个Phase [来源: SKILL.md:339]

### Phase 2: 知识采集

- **执行者**: planner 子Agent
- **输入**: 需求清单.md + 项目代码库 + EXTENSION_DOCS.requirement（可选）
- **输出**: `代码阅读报告.md` + `知识摘要.md` + `log/phase2-知识采集.md`
- **职责**: GitNexus索引（可选）→ 六步下钻法（模块入口→领域模型→核心业务→数据访问→边界交互→配置基础设施）→ 知识库查阅
- **设计理念体现**: 文档驱动交接 — 知识摘要作为P3的设计依据; GitNexus赋能 — 知识图谱辅助代码分析

### Phase 3: 系分编写

- **执行者**: designer 子Agent
- **输入**: 需求清单.md + 代码阅读报告.md + 知识摘要.md + EXTENSION_DOCS.design（可选）
- **输出**: `系分文档.md` + `需求追踪矩阵.md` + `log/phase3-系分编写.md`
- **职责**: 生成骨架→用户确认→生成需求追踪矩阵→填充内容（遵循架构约束6 MUST + 5 MUST NOT）→自检
- **关键产出**: 需求追踪矩阵 — 每条需求映射到设计章节、接口、数据变更
- **设计理念体现**: 文档驱动交接; 按需求隔离; 需求驱动全流程

### Phase 4: 系分评审（并行）

- **执行者**: 3个并行 reviewer 子Agent [来源: SKILL.md:162-173]
  - **design-reviewer** — 架构合理性（分层/耦合/扩展性/约束遵循/过度设计）
  - **performance-reviewer** — 性能容量（索引/N+1/缓存/锁竞争/容量预估）
  - **req-completeness-reviewer** — 需求完整度（逐条核对R01-RNN，确认每条都有设计覆盖）
- **输入**: 系分文档.md + 需求追踪矩阵.md + 需求清单.md + 代码阅读报告.md + 评审检查清单.md + EXTENSION_DOCS.review（可选）
- **输出**: `log/phase4-评审-架构.md` + `log/phase4-评审-性能.md` + `log/phase4-评审-需求完整度.md`
- **Pass Criteria**: 0 CRITICAL + 0 HIGH + 需求覆盖100% [来源: SKILL.md:171]
- **FAIL时**: 修正循环 — Resume designer 定点修复 → Resume reviewers 重新评审，≤ MAX_ITERATION 轮
- **设计理念体现**: Resume优于新建; 经验库累积（修正中沉淀的模式写入经验库）

**注意**: SKILL.md L88 提到 Phase 4 有"4个评审"，但 L162-170 角色表只列3个。执行日志规范中 Agent注册表示例列了4个（含 security-reviewer），但系分评审执行指南只定义3个角色。此处存在不一致。产出目录结构中也列了 `phase4-评审-安全.md` [来源: SKILL.md:247]。Dashboard 应以执行指南定义的 3 个角色为准，但兼容 4 个文件。

### Phase 5: 编码实现

- **执行者**: coder 子Agent（可 Resume）
- **输入**: 需求清单.md + 系分文档.md + 需求追踪矩阵.md + 经验库.md（可选但必读如存在）+ EXTENSION_DOCS.coding（可选）
- **输出**: 代码文件 + 测试文件 + 需求清单状态更新（⏳→💻）+ `log/phase5-编码.md`
- **职责**: 按 BATCH_SIZE 分批实现，每条需求 ≥1 个测试，严格按系分实现
- **设计理念体现**: 文档驱动交接 — 严格按系分实现; 经验库累积 — 必读经验库避免重犯

### Phase 6: 代码 CR（并行）

- **执行者**: 3个并行 reviewer 子Agent [来源: SKILL.md:192-201]
  - **code-quality-reviewer** — 代码质量（可读性/复杂度/重复/错误处理/测试质量/代码军规）
  - **code-security-reviewer** — 安全漏洞（注入/XSS/越权/硬编码密钥/日志泄露）
  - **code-req-reviewer** — 需求实现完整度（逐条在代码中验证R01-RNN）
- **输入**: 需求清单.md + 需求追踪矩阵.md + 系分文档.md + 评审检查清单.md + EXTENSION_DOCS.review + EXTENSION_DOCS.coding + 代码/测试文件
- **输出**: `log/phase6-CR-质量.md` + `log/phase6-CR-安全.md` + `log/phase6-CR-需求实现度.md`
- **Pass Criteria**: 0 CRITICAL + 0 HIGH + 需求实现100%
- **FAIL时**: Resume coder 修复 → Resume reviewers，≤ MAX_ITERATION 轮

### Phase 7: 测试验证

- **执行者**: tester 子Agent
- **输入**: 代码+测试文件 + 需求清单.md + EXTENSION_DOCS.testing + GitNexus索引
- **输出**: `log/phase7-测试.md`，需求清单状态更新（💻→✅）
- **职责**: 运行测试+lint+typecheck → GitNexus影响面分析 → 逐条需求确认测试通过
- **设计理念体现**: GitNexus赋能 — 影响面分析确保变更范围全覆盖

### Phase 8: 交付报告

- **执行者**: 主Agent（非子Agent）
- **输入**: 需求清单.md + 各阶段日志 + git diff
- **输出**: `交付报告.md` + `log/执行摘要.md`
- **职责**: 汇总需求完成率、产出物清单、变更文件、迭代统计、残留风险、经验沉淀、commit message 建议
- **设计理念体现**: 主Agent纯调度 — P8 是主Agent唯一的直接产出（非子Agent）

## 风险和复杂点

### 1. P4 角色数量不一致

SKILL.md L88 说"4个评审"，但角色表只列3个。执行日志规范的 Agent 注册表示例列了4个角色（含 security-reviewer），目录结构也列了 `phase4-评审-安全.md`。Dashboard 应灵活处理：扫描 `phase4-评审-*.md` 文件动态展示，而非硬编码3个。

### 2. Log 文件可能不存在

流水线运行中，某些 Phase 尚未开始时对应的 log 文件不存在。后端扫描需对此做容错处理——文件不存在时返回"未开始"状态。

### 3. 迭代日志编号动态增长

`phase4-迭代-1.md`、`phase4-迭代-2.md`... 文件名中的编号是动态的。后端需通过模式匹配 `phase{N}-迭代-{M}.md` 扫描所有迭代文件。

### 4. 评审报告解析——判定行提取

主Agent用 Grep 提取 `## 判定：PASS/FAIL`。Dashboard 后端也需要同样的解析逻辑来确定各评审的通过/失败状态。

### 5. 需求状态分散在多处

需求状态（⏳→📐→💻→✅→⚠️）记录在需求清单.md 的表格中。后端需要解析 Markdown 表格提取每条需求的当前状态。

### 6. Markdown 渲染零依赖

用户要求纯 JS 手写 Markdown 解析器，需支持：标题、列表、代码块、表格、链接。这是前端开发的主要复杂点。

### 7. 并行 Agent 的 log 文件同时生成

P4 和 P6 的并行 reviewer 各自写独立文件（防冲突设计 [来源: reference/执行日志规范.md:86-101]），后端需同时扫描多个文件展示并行状态。

### 8. 两种显示模式切换

无运行数据时展示设计理念模式（R11），有数据时展示实时状态。需要前端判断后端返回数据是否为空来切换模式。
