# 评审检查清单

> Phase 4（系分评审）和 Phase 6（代码CR）的各 reviewer 子Agent 使用对应章节。

---

## §系分-架构 — design-reviewer

- [ ] 分层清晰，模块边界合理
- [ ] 无循环依赖或不合理跨层调用
- [ ] 扩展性：未来变化时改动范围可控
- [ ] 一致性：命名/风格/模式与现有系统一致
- [ ] 复杂度：有没有更简单的方案？是否过度设计？
- [ ] 接口粒度：不过粗(God API)不过细(chatty API)
- [ ] MUST 约束逐条核对（见架构约束.md）
- [ ] MUST NOT 约束逐条核对

---

## §系分-性能 — performance-reviewer

- [ ] 热路径有性能优化
- [ ] 数据库查询走索引，无N+1问题
- [ ] 缓存设计：穿透/击穿/雪崩防护
- [ ] 并发：锁竞争和锁粒度合理
- [ ] 容量预估：QPS / 数据量 / 连接数
- [ ] 可异步的已设计为异步
- [ ] 无不合理的循环单条处理

---

## §需求完整度 — req-completeness-reviewer

**Phase 4 系分评审时使用。逐条核对，不允许遗漏。**

对 requirements.md 中的每条需求 (R01, R02, ...):

1. [ ] 需求是否有对应的设计章节？
2. [ ] 接口设计是否覆盖该需求功能点？
3. [ ] 数据模型是否支持该需求的数据需求？
4. [ ] 边界场景是否考虑？（并发、空数据、重复提交）
5. [ ] 异常流程是否有设计？（失败重试、补偿）

**判定**:
- 未覆盖 → **CRITICAL**
- 部分覆盖（标明缺失部分） → **HIGH**

**输出格式**:
```markdown
### 需求覆盖度

| 需求ID | 描述 | 设计章节 | 接口 | 数据 | 边界 | 判定 |
|--------|------|---------|------|------|------|------|
| R01 | ... | §3.1 | ✅ | ✅ | ✅ | PASS |
| R05 | ... | 无 | ❌ | ❌ | - | CRITICAL |
```

---

## §代码-质量 — code-quality-reviewer

- [ ] 可读性：命名清晰、逻辑自明
- [ ] 复杂度：函数<50行、文件<800行、嵌套<4层
- [ ] 重复：可提取的公共逻辑
- [ ] 错误处理：全面、信息友好
- [ ] 测试质量：覆盖率、边界case、断言充分
- [ ] 一致性：与现有代码风格一致
- [ ] 扩展文档遵循：如有代码军规等，逐条核对

---

## §代码-安全 — code-security-reviewer

- [ ] SQL/NoSQL 注入
- [ ] XSS（反射型、存储型）
- [ ] 硬编码密钥/token/密码
- [ ] 越权访问（水平/垂直）
- [ ] 日志泄露敏感信息
- [ ] 不安全的反序列化
- [ ] 路径遍历
- [ ] SSRF

---

## §需求实现度 — code-req-reviewer

**Phase 6 代码CR时使用。在代码中逐条验证需求实现。**

对 requirements.md 中的每条需求:

1. [ ] 接口是否已实现？（路由注册、handler 存在）
2. [ ] 业务逻辑是否匹配设计？（非空壳实现）
3. [ ] 验收标准中的条件是否有代码覆盖？
4. [ ] 是否有对应的测试？测试是否覆盖验收标准？

**判定**:
- 未实现 → **CRITICAL**
- 实现不完整（缺测试、缺校验） → **HIGH**

**输出格式**:
```markdown
### 需求实现度

| 需求ID | 描述 | 接口 | 逻辑 | 测试 | 判定 |
|--------|------|------|------|------|------|
| R01 | ... | ✅ | ✅ | ✅ | PASS |
| R03 | ... | ✅ | ⚠️ | ❌ | HIGH |
```

---

## 通用输出格式

所有 reviewer 的问题清单：

```markdown
| # | 严重度 | 视角 | 位置 | 问题 | 建议 | 关联需求 |
|---|--------|------|------|------|------|----------|
```

严重度: CRITICAL > HIGH > MEDIUM > LOW

## 汇总 Pass Criteria

| 条件 | 结果 |
|------|------|
| 0 CRITICAL + 0 HIGH + 需求覆盖/实现100% | **PASS** |
| 存在 CRITICAL/HIGH 或需求未覆盖 | **FAIL** → 修正循环 |
