# 执行日志规范

> 主Agent 维护 `{需求目录}/log/执行日志.md`。子Agent 各写独立 log 文件，不关心本日志。

## 时间格式

`YYYY-MM-DD HH:MM:SS`

## 事件类型（枚举，与 server.py `AGENT_EVENTS` 一致）

| 事件 | 含义 | Dashboard 状态 |
|------|------|---------------|
| 启动 | 首次启动 Agent | running |
| 完成 | 非评审 Agent 正常完成 | pass |
| PASS | 评审 Agent 判定通过 | pass |
| FAIL | 评审 Agent 判定不通过 | fail |
| Resume | 修正循环 Resume 同一 Agent | running |
| 降级新建 | SendMessage 不可用，新建 Agent 替代 | running |

**只能使用以上枚举值，不可自创事件名。Dashboard 通过此枚举映射确定 Phase 状态。**

## Agent 名称（枚举，与 server.py `PHASE_CONFIG` 一致）

| Phase | Agent名称 | 角色 |
|-------|----------|------|
| P1 | P1-需求访谈员 | interviewer |
| P2 | P2-知识采集员 | collector |
| P3 | P3-系分设计师 | designer |
| P4 | P4-架构评审员 | design-reviewer |
| P4 | P4-性能评审员 | performance-reviewer |
| P4 | P4-需求完整度评审员 | req-completeness-reviewer |
| P5 | P5-编码工程师 | coder |
| P6 | P6-代码质量评审员 | code-quality-reviewer |
| P6 | P6-安全评审员 | code-security-reviewer |
| P6 | P6-需求实现度评审员 | code-req-reviewer |
| P7 | P7-测试工程师 | tester |
| P8 | P8-主Agent | 主Agent（交付报告） |

**只能使用以上名称，不可自创。Dashboard 通过 Agent名称 匹配 Phase 和角色。**

## 执行日志模板

```markdown
# 执行日志 — {feature-name}

## 项目信息
- 启动时间: {datetime}
- 需求来源: {source}
- 配置: BATCH_SIZE={N}, MAX_ITERATION={N}

## Agent 执行记录

> 仅追加，不可修改已有行。主 Agent 负责记录，子 Agent 不关心本表。
> Dashboard 从本表派生所有 Phase 状态，遗漏事件将导致状态不正确。

| # | 时间 | Phase | Agent名称 | agentId | 事件 | 备注 |
|---|------|-------|----------|---------|------|------|
```

## 日志规则

1. 每次 Agent 事件追加新行，序号递增
2. **仅追加，不可修改已有行**
3. 事件列只能使用上述枚举值（启动/完成/PASS/FAIL/Resume/降级新建）
4. Agent名称列只能使用上述枚举名称
5. 评审结果用 PASS/FAIL，非评审用 完成
6. 修正循环: Resume → (修复) → 完成/PASS/FAIL，每步各追加一行
7. agentId 从 Agent 工具返回值提取
8. 备注中评审统计格式: `NC/NH/NM/NL`（如 `0C/1H/2M/0L`）

每个子Agent写独立文件，防止并行写入冲突：

|---------|------|
| interviewer | `log/phase1-需求澄清.md` |
| collector | `log/phase2-知识采集.md` |
| designer | `log/phase3-系分编写.md` |
| design-reviewer | `log/phase4-评审-架构.md` |
| performance-reviewer | `log/phase4-评审-性能.md` |
| req-completeness-reviewer | `log/phase4-评审-需求完整度.md` |
| coder | `log/phase5-编码.md` |
| code-quality-reviewer | `log/phase6-CR-质量.md` |

| 迭代修正 | `log/phase{N}-迭代-{轮次}.md` |

## 执行摘要

全流程完成后生成 `log/执行摘要.md`:

```markdown
# 执行摘要

| 阶段 | 日志文件 | 结果 |
|------|---------|------|
| Phase 1 需求澄清 | 需求清单.md | {N}条需求 |
| Phase 2 知识采集 | log/phase2-知识采集.md | 完成 |
| Phase 3 系分编写 | log/phase3-系分编写.md | 完成 |
| Phase 4 系分评审 | log/phase4-评审-*.md | PASS (第{N}轮) |
| Phase 5 编码实现 | log/phase5-编码.md | {N}条需求已编码 |
| Phase 6 代码CR | log/phase6-CR-*.md | PASS (第{N}轮) |
| Phase 7 测试验证 | log/phase7-测试.md | {N}/{M} passed |
| Phase 8 交付报告 | 交付报告.md | 完成率 XX% |
```
