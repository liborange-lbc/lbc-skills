# 系分评审执行指南

> Phase 4 启动 3 个并行评审子Agent，各自独立评审，互不干扰。

## 角色分工

| 角色 | 视角 | 核心关注 |
|------|------|----------|
| design-reviewer | 架构合理性 | 分层、耦合、扩展性、约束遵循、是否过度设计 |
| performance-reviewer | 性能容量 | 索引、N+1、缓存策略、锁竞争、容量预估 |
| req-completeness-reviewer | 需求完整度 | **逐条核对**需求清单，确认每条都有设计覆盖 |

## 输入

每个评审子Agent启动时，主Agent通过 prompt 注入以下内容（见 SKILL.md §子Agent Prompt 组装规范）：

- **需读取的文件路径**（子Agent用 Read 工具自行读取）:
  1. `{需求目录}/系分文档.md` — 评审对象
  2. `{需求目录}/需求追踪矩阵.md` — 需求-设计映射
  3. `{需求目录}/需求清单.md` — 需求条目列表（req-completeness-reviewer 必读）
  4. `{需求目录}/代码阅读报告.md` — 现有代码上下文
- **主Agent直接注入prompt的内容**:
  5. 评审检查清单（对应角色章节） — 由主Agent从skill目录读取后注入
  6. 架构约束（仅 design-reviewer） — 同上
  7. EXTENSION_DOCS.review 中的文档路径（如有）

## 各角色执行流程

### design-reviewer — 架构合理性

1. 读取系分文档全文
2. 按 评审检查清单（已注入prompt） §系分-架构 逐条核对
3. 读取架构约束（已注入prompt），逐条验证 MUST / MUST NOT
4. 如有 EXTENSION_DOCS.review 扩展文档，按其中的规则追加检查
5. 产出评审报告，写入 `log/phase4-评审-架构.md`

### performance-reviewer — 性能容量

1. 读取系分文档全文 + 代码阅读报告（了解现有性能约束）
2. 按 评审检查清单（已注入prompt） §系分-性能 逐条核对
3. 重点关注：数据库查询设计、缓存策略、容量预估、异步化设计
4. 产出评审报告，写入 `log/phase4-评审-性能.md`

### req-completeness-reviewer — 需求完整度

1. 读取需求清单（R01-RNN），逐条记录
2. 读取需求追踪矩阵，检查每条需求的映射
3. 读取系分文档，按 评审检查清单（已注入prompt） §需求完整度 **逐条验证**：
   - 该需求是否有对应的设计章节？
   - 接口设计是否覆盖该需求功能点？
   - 数据模型是否支持该需求的数据需求？
   - 边界场景是否考虑？
   - 异常流程是否有设计？
4. 未覆盖 → CRITICAL，部分覆盖 → HIGH
5. 产出评审报告，写入 `log/phase4-评审-需求完整度.md`

## 产出格式

每个角色的评审报告统一格式：

```markdown
# 系分评审报告 — {角色名}

## 判定：PASS / FAIL

## 统计
- CRITICAL: N 项
- HIGH: N 项
- MEDIUM: N 项
- LOW: N 项

## 需求覆盖度（仅 req-completeness-reviewer）
| 需求ID | 描述 | 设计章节 | 接口 | 数据 | 边界 | 判定 |
|--------|------|---------|------|------|------|------|

## 问题清单
| # | 严重度 | 位置 | 问题 | 修改建议 | 关联需求 |
|---|--------|------|------|----------|----------|
```

**关键**: 报告第一行必须是 `## 判定：PASS` 或 `## 判定：FAIL`，主Agent用 Grep 提取此行。

## 行为约束

1. **只读不改** — 评审子Agent 绝不修改系分文档，只产出评审报告
2. **独立评审** — 不读其他 reviewer 的报告，避免偏见
3. **具体建议** — 每个问题必须给出"改哪里、改成什么"的具体建议
4. **标注关联需求** — 如果问题关联某条需求，填写需求ID
5. **扩展文档优先** — 如有扩展评审标准，其中的规则优先于默认 checklist
