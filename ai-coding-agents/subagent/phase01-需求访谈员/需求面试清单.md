# 需求澄清执行指南

> interviewer 子Agent 在 Phase 1 使用。目标：消除歧义，产出可追踪的原子级需求清单。

## Input

| 文件 | 必须 | 说明 |
|------|------|------|
| 用户的需求描述 | 是 | 主Agent转发的用户原始需求 |
| EXTENSION_DOCS.requirement | 否 | 产品PRD、业务流程图等（补充理解） |

## Execution

### Step 1: 8维度面试

用 AskUserQuestion 逐项确认：

| # | 维度 | 问题 | 目的 |
|---|------|------|------|
| 1 | 业务背景 | 为什么做？解决什么业务问题？ | 避免偏离 |
| 2 | 核心功能 | 具体做什么？逐条列出 | 明确范围 |
| 3 | 影响范围 | 涉及哪些模块/服务/团队？ | 识别依赖 |
| 4 | 用户角色 | 谁用？几种角色？权限差异？ | 鉴权设计 |
| 5 | 数据流向 | 数据从哪来、到哪去、怎么变？ | 接口和存储 |
| 6 | 非功能需求 | QPS预估？延迟？数据量级？ | 性能和容量 |
| 7 | 约束条件 | deadline？兼容性？不能动的部分？ | 设计边界 |
| 8 | 验收标准 | 怎么算做完了？ | 自检依据 |

### 追问策略

- 模糊回答 → 给 2-3 个选项让用户选
- "你看着办" → 给出假设让用户确认
- 矛盾回答 → 指出矛盾，请用户裁决

### Step 2: 需求拆分

将面试结果拆解为**原子级需求条目**：

**粒度标准**:
- 每条需求可以用**一个测试用例**验证
- 每条需求可以在**一次编码迭代**中完成
- 每条需求有**明确的验收标准**

**拆分示例**:

❌ 太粗: "实现用户注册功能"

✅ 合适:
- R01: 用户输入手机号，系统发送6位验证码（有效期5分钟）
- R02: 用户提交验证码，校验通过后创建账户
- R03: 注册时校验手机号唯一性，重复则返回错误提示
- R04: 注册成功后自动登录，返回JWT token
- R05: 验证码60秒内不可重发（防刷）

### 编号规则

- 功能需求: `R{NN}` — R01, R02, ...
- 非功能需求: `NFR{NN}` — NFR01 性能, NFR02 安全
- 按优先级排序: P0 (必须) > P1 (应该) > P2 (可选)

### Step 3: 自检

- [ ] 每条需求有明确的验收标准
- [ ] 无重叠需求（两条需求描述同一件事）
- [ ] 无遗漏（8维度面试覆盖的所有功能点都已拆分）
- [ ] 非功能需求独立列出（性能、安全等）

## Output

产出**一个文件**，写入 `{需求目录}/需求清单.md`：

```markdown
# 需求确认清单 — {feature-name}

## 业务背景
{用户确认的背景描述}

## 需求清单

| ID | 需求描述 | 验收标准 | 优先级 | 状态 |
|----|---------|---------|--------|------|
| R01 | ... | ... | P0 | ⏳ |
| R02 | ... | ... | P0 | ⏳ |

## 非功能需求

| ID | 描述 | 指标 |
|----|------|------|
| NFR01 | 接口响应时间 | ≤ 200ms (P95) |

## 约束条件
- {用户明确的约束}

## 确认
- [ ] 用户已确认以上需求清单完整且准确
```

### 文档质量要求

- [ ] 每条需求都有验收标准（不允许"待定"）
- [ ] 每条需求都有优先级
- [ ] 需求总数 ≥ 3（如果只有1-2条，说明拆分不够细）
- [ ] 非功能需求至少包含性能指标
- [ ] "确认"章节的 checkbox 必须由主Agent转交用户勾选后才进入 Phase 2

### 日志

写入 `{需求目录}/log/phase1-需求澄清.md`（记录面试过程和关键决策）
