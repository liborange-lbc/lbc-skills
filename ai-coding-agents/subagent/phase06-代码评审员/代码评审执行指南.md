# 代码评审执行指南

> Phase 6 启动 3 个并行评审子Agent，各自独立 CR，互不干扰。

## 角色分工

| 角色 | 视角 | 核心关注 |
|------|------|----------|
| code-quality-reviewer | 代码质量 | 可读性、复杂度、重复、错误处理、测试质量、代码军规 |
| code-security-reviewer | 安全漏洞 | 注入、XSS、越权、硬编码密钥、日志泄露 |
| code-req-reviewer | 需求实现完整度 | **逐条核对**需求清单，确认每条需求有代码实现和测试 |

## 输入

每个评审子Agent启动时，主Agent传入以下文件路径：

```
1. {需求目录}/需求清单.md           — 需求条目（code-req-reviewer 必读）
2. {需求目录}/需求追踪矩阵.md      — 需求→接口→文件的映射
3. {需求目录}/系分文档.md           — 设计依据（对照实现是否匹配设计）
4. reference/评审检查清单.md        — 各角色的 checklist
5. EXTENSION_DOCS.review 中的文档  — 扩展评审标准（如有）
6. EXTENSION_DOCS.coding 中的文档  — 代码军规等（code-quality-reviewer 必读）
```

注意：子Agent 需要自行读取项目中的代码文件和测试文件（通过追踪矩阵定位）。

## 各角色执行流程

### code-quality-reviewer — 代码质量

1. 读取需求追踪矩阵，定位本次变更涉及的所有代码文件
2. 逐文件阅读代码，按 [reference/评审检查清单.md](../../reference/评审检查清单.md) §代码-质量 核对：
   - 可读性：命名清晰、逻辑自明
   - 复杂度：函数<50行、文件<800行、嵌套<4层
   - 重复：可提取的公共逻辑
   - 错误处理：全面、信息友好
   - 测试质量：覆盖率、边界case、断言充分
   - 一致性：与现有代码风格一致
3. 如有 EXTENSION_DOCS.coding（代码军规），**逐条核对**代码是否遵循
4. 产出评审报告，写入 `log/phase6-CR-质量.md`

### code-security-reviewer — 安全漏洞

1. 读取需求追踪矩阵，定位本次变更涉及的所有代码文件
2. 逐文件阅读代码，按 [reference/评审检查清单.md](../../reference/评审检查清单.md) §代码-安全 核对：
   - SQL/NoSQL 注入
   - XSS（反射型、存储型）
   - 硬编码密钥/token/密码
   - 越权访问（水平/垂直）
   - 日志泄露敏感信息
   - 不安全的反序列化
   - 路径遍历、SSRF
3. 产出评审报告，写入 `log/phase6-CR-安全.md`

### code-req-reviewer — 需求实现完整度

1. 读取需求清单（R01-RNN），逐条记录需求描述和验收标准
2. 读取需求追踪矩阵，找到每条需求对应的接口/模块/文件
3. **在代码中逐条验证**（按 [reference/评审检查清单.md](../../reference/评审检查清单.md) §需求实现度）：
   - 接口是否已实现？（路由注册、handler 存在）
   - 业务逻辑是否匹配设计？（非空壳实现）
   - 验收标准中的条件是否有代码覆盖？
   - 是否有对应的测试？测试是否覆盖验收标准？
4. 未实现 → CRITICAL，实现不完整 → HIGH
5. 产出评审报告，写入 `log/phase6-CR-需求实现度.md`

## 产出格式

每个角色的评审报告统一格式：

```markdown
# 代码CR报告 — {角色名}

## 判定：PASS / FAIL

## 统计
- CRITICAL: N 项
- HIGH: N 项
- MEDIUM: N 项
- LOW: N 项

## 需求实现度（仅 code-req-reviewer）
| 需求ID | 描述 | 接口 | 逻辑 | 测试 | 判定 |
|--------|------|------|------|------|------|

## 问题清单
| # | 严重度 | 文件:行号 | 问题 | 修改建议 | 关联需求 |
|---|--------|----------|------|----------|----------|
```

**关键**: 报告第一行必须是 `## 判定：PASS` 或 `## 判定：FAIL`，主Agent用 Grep 提取此行。

## 行为约束

1. **只读不改** — 评审子Agent 绝不修改代码文件，只产出评审报告
2. **独立评审** — 不读其他 reviewer 的报告，避免偏见
3. **精确定位** — 问题必须标注 `文件:行号`，不能只说"某处有问题"
4. **具体建议** — 每个问题给出"改哪里、改成什么"的具体建议
5. **标注关联需求** — 如果问题关联某条需求，填写需求ID
6. **扩展文档优先** — 代码军规中的规则优先于默认 checklist

## 与 Phase 4 系分评审的区别

| 维度 | Phase 4 系分评审 | Phase 6 代码CR |
|------|-----------------|---------------|
| 评审对象 | 系分文档（设计） | 代码文件（实现） |
| 定位方式 | 章节号 §3.1 | 文件:行号 src/api.ts:42 |
| 需求核对 | 设计是否覆盖需求 | 代码是否实现需求 + 测试是否覆盖 |
| 安全视角 | Phase 4 不做安全评审 | Phase 6 包含安全评审 |
