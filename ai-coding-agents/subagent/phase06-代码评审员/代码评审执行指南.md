# 代码评审执行指南

> Phase 6 启动 3 个并行评审子Agent，各自独立 CR，互不干扰。采用 SDD 模式结构化审查。

## 严重性等级

| 等级 | 名称 | 对应原有等级 | 描述 | 处理方式 |
|------|------|-------------|------|----------|
| **P0** | 阻塞 | CRITICAL | 功能性不符、安全漏洞、严重可靠性问题、bug隐患 | 必须阻止合并 |
| **P1** | 推荐 | HIGH | 安全隐患、可靠性隐患、严重影响可读性 | 合并前应修复 |
| **P2** | 参考 | MEDIUM | 可靠性建议、代码风格 | 可选改进 |

## 角色分工

| 角色 | 视角 | 核心关注 | 对应 SDD 检查维度 |
|------|------|----------|------------------|
| code-quality-reviewer | 代码质量 | 可读性、复杂度、重复、错误处理、测试质量、代码军规 | Step 3 可读性 + Step 4 可靠性(军规) + Step 5 自定义 |
| code-security-reviewer | 安全漏洞 | 注入、XSS、越权、硬编码密钥、日志泄露、Bug模式 | Step 4 安全 + Step 4 Bug模式 |
| code-req-reviewer | 需求实现完整度 | **逐条核对**需求清单，确认每条需求有代码实现和测试 | Step 2 功能性检查(REQ核对) |

## 输入

每个评审子Agent启动时，主Agent通过 prompt 注入以下内容（见 SKILL.md §子Agent Prompt 组装规范）：

- **需读取的文件路径**（子Agent用 Read 工具自行读取）:
  1. `{需求目录}/需求清单.md` — 需求条目（code-req-reviewer 必读）
  2. `{需求目录}/需求追踪矩阵.md` — 需求→接口→文件的映射
  3. `{需求目录}/系分文档.md` — 设计依据（对照实现是否匹配设计）
  4. 项目中的代码文件和测试文件（通过追踪矩阵定位）
- **主Agent直接注入prompt的内容**:
  5. 评审检查清单（对应角色章节） — 由主Agent从skill目录读取后注入
  6. EXTENSION_DOCS.review 中的文档路径（如有）
  7. EXTENSION_DOCS.coding 中的文档路径（code-quality-reviewer 必读，如有）

## 执行流程

### 预检：确定审查范围（所有角色通用）

1. 读取需求追踪矩阵，定位本次变更涉及的所有代码文件（**直接变更文件**）
2. **GitNexus 影响面扩展**（code-quality-reviewer 和 code-security-reviewer 执行）：
   ```bash
   # 检测变更影响的符号和执行流
   gitnexus detect-changes --scope all --base-ref {base-branch} -r {project-name}
   # 对关键变更符号查上下游依赖（深度2层）
   gitnexus impact {ChangedSymbol} --direction both --depth 2 -r {project-name}
   ```
   - 将间接影响的文件加入执行队列，标记为 `⚡ 影响面扩展`（与直接变更文件区分）
   - **影响面扩展文件只做可靠性和安全检查**，不做可读性检查（非本次变更代码）
   - **降级方案**：GitNexus 不可用时退回仅审查追踪矩阵中的文件，报告中标注 "影响面分析: N/A"
3. 构建**执行队列**：展开为一行一个仓库内相对路径，按 TECH_STACK 筛选源码文件（Java 项目为 `*.java`）
4. **非源码文件处理**：GitNexus 返回的非源码文件（如 `*.yml`, `*.xml`, `*.sql`）不进入执行队列，但在报告"影响面分析"章节列出供人工关注
5. 跳过项标注 `跳过`，禁止以 glob 代替展开

### 自动化预扫（code-quality-reviewer 和 code-security-reviewer 执行）

在进入逐文件 LLM 审查**之前**，必须先执行 `references/script/scan-all-rules.sh`（脚本由主Agent注入路径或内容）：

- 对本次变更涉及的目录或文件路径跑一次统一扫描
- 将脚本完整输出（含 `path:line`）写入检查清单对应备注
- 据此先把脚本已覆盖规则标为 `❌/⚠️`（或确认 `✅`）
- 随后由 LLM 完成脚本未覆盖项的人工复核

### code-quality-reviewer — 代码质量

**逐文件串行审查**，每个文件按顺序完成：

1. **可读性检查**（对照 [references/readability-checklist.md](references/readability-checklist.md) A1-A7）：
   - 命名清晰、逻辑自明
   - 复杂度：函数<50行、文件<800行、嵌套<4层
   - 重复：可提取的公共逻辑
   - 一致性：与现有代码风格一致
   - 违规标 P2（一般风格）或 P1（明显影响可读性/可维护性）

2. **可靠性检查**（对照 [references/reliability-checklist.md](references/reliability-checklist.md)）：
   - 超时/重试/限流
   - 资源释放
   - 事务边界
   - 并发与幂等
   - 边界条件
   - 灰度/监控/应急
   - 等级以清单行内 P0/P1/P2 为准

3. **Bug模式检查**（对照 [references/bug-pattern-checklist.md](references/bug-pattern-checklist.md)）：
   - 阿里巴巴 Java 代码缺陷类规则（120条，ID为 B*/M*/I*）
   - 脚本已覆盖的直接引用扫描结果
   - 脚本未覆盖项由 LLM 逐条核对，命中须写**规则ID + `path:line`**
   - 等级映射：Blocker→P0、Major→P1、Info→P2

4. **自定义扩展检查**（对照 [references/customized-checklist.md](references/customized-checklist.md)）：
   - 按团队或项目特定要求追加检查项
   - 若该文件为空或全为示例项，标 `N/A(未启用自定义规则)`

5. 如有 EXTENSION_DOCS.coding（代码军规），**逐条核对**代码是否遵循
6. 更新执行队列状态（`✅ 已审` / `⚠️ 已审有问题`）
7. 产出评审报告，写入 `log/phase6-CR-质量.md`

### code-security-reviewer — 安全漏洞

**逐文件串行审查**，每个文件完成：

1. **安全检查**（对照 [references/security-checklist.md](references/security-checklist.md)）：
   - 认证/授权
   - 输入校验
   - 密钥泄露
   - 依赖安全
   - SQL/NoSQL 注入
   - XSS（反射型、存储型）
   - 硬编码密钥/token/密码
   - 越权访问（水平/垂直）
   - 日志泄露敏感信息
   - 不安全的反序列化
   - 路径遍历、SSRF
   - 等级以清单行内 P0/P1 为准

2. **Bug模式-安全相关**（对照 [references/bug-pattern-checklist.md](references/bug-pattern-checklist.md) 中安全相关条目）：
   - 与 code-quality-reviewer 的 Bug模式检查互补
   - 聚焦安全类缺陷规则

3. 更新执行队列状态
4. 产出评审报告，写入 `log/phase6-CR-安全.md`

### code-req-reviewer — 需求实现完整度

**逐需求核对**（SDD Step 2 功能性检查）：

1. 读取需求清单（R01-RNN），逐条记录需求描述和验收标准
2. 读取需求追踪矩阵，找到每条需求对应的接口/模块/文件
3. **REQ 来源约束**：每个 REQ 必须来自系分文档/需求清单原文，记录原文摘录、章节号或标题锚点
4. **关联文件约束**：每个 REQ 必须绑定至少一个变更文件；若暂无法绑定，需明确说明原因
5. **在代码中逐条验证**：
   - 接口是否已实现？（路由注册、handler 存在）
   - 业务逻辑是否匹配设计？（非空壳实现）
   - 验收标准中的条件是否有代码覆盖？
   - 是否有对应的测试？测试是否覆盖验收标准？
6. **P0 证据约束**：若判断不符合 REQ，必须同时给出 **spec 证据**（系分文档引文）与 **代码证据**（`path:line` / 测试 / 接口行为）
7. 未实现 → P0/CRITICAL，实现不完整 → P1/HIGH
8. 产出评审报告，写入 `log/phase6-CR-需求实现度.md`

## 收口验证

各角色队列完成后：

- 执行队列 `⬜ 待审` 必须为零（跳过项除外）
- 报告审查范围文件数与已审队列一致
- code-req-reviewer 核对 REQ 级勾选与逐文件结论一致

## 产出格式

每个角色的评审报告统一格式：

```markdown
# 代码CR报告 — {角色名}

## 判定：PASS / FAIL

## 统计
- CRITICAL: N 项
- HIGH: N 项
- MEDIUM: N 项
- LOW: N 项

## 执行队列
| # | 文件 | 来源 | 状态 |
|---|------|------|------|
| 1 | src/service/XxxService.java | 直接变更 | ✅ 已审 |
| 2 | src/controller/XxxController.java | 直接变更 | ⚠️ 已审有问题 |
| 3 | src/service/YyyService.java | ⚡ 影响面扩展 | ✅ 已审(仅可靠性+安全) |

## 影响面分析（GitNexus）
- detect-changes 发现 N 个受影响符号
- 扩展文件数: M 个
- 影响面分析状态: 已完成 / N/A(GitNexus不可用)

## 需求实现度（仅 code-req-reviewer）
| 需求ID | 描述 | 接口 | 逻辑 | 测试 | 判定 | spec证据 |
|--------|------|------|------|------|------|----------|

## 问题清单
| # | 严重度 | 规则ID | 文件:行号 | 问题 | 修改建议 | 关联需求 |
|---|--------|--------|----------|------|----------|----------|

## 修复任务列表
- [ ] {待修复项1 — 从问题清单汇总}
- [ ] {待修复项2}
（若无待办，写明「无待修复项」）
```

**关键**: 报告第一行必须是 `## 判定：PASS` 或 `## 判定：FAIL`，主Agent用 Grep 提取此行。

## 行为约束

1. **只读不改** — 评审子Agent 绝不修改代码文件，只产出评审报告
2. **独立评审** — 不读其他 reviewer 的报告，避免偏见
3. **精确定位** — 问题必须标注 `文件:行号`，不能只说"某处有问题"
4. **具体建议** — 每个问题给出"改哪里、改成什么"的具体建议
5. **标注关联需求** — 如果问题关联某条需求，填写需求ID
6. **标注规则ID** — Bug模式问题必须写规则ID（B*/M*/I*），便于追踪
7. **扩展文档优先** — 代码军规中的规则优先于默认 checklist
8. **证据约束** — P0 问题必须同时提供 spec 证据和代码证据

## 与 Phase 4 系分评审的区别

| 维度 | Phase 4 系分评审 | Phase 6 代码CR |
|------|-----------------|---------------|
| 评审对象 | 系分文档（设计） | 代码文件（实现） |
| 定位方式 | 章节号 §3.1 | 文件:行号 src/api.ts:42 |
| 需求核对 | 设计是否覆盖需求 | 代码是否实现需求 + 测试是否覆盖 |
| 安全视角 | Phase 4 不做安全评审 | Phase 6 包含安全评审 |
| 自动化预扫 | 无 | scan-all-rules.sh 预扫 |
| Bug模式 | 无 | 120条阿里巴巴Java缺陷规则 |
| 影响面分析 | 无 | GitNexus detect-changes + impact 扩展审查范围 |
