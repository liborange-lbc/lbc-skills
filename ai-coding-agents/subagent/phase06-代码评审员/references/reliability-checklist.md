# 可靠性检查（参考 蚂蚁编码军规 + 军规 V2.0 摘录）

> 按下方 **ID** 逐类扫描变更代码；命中则输出 `[等级] ID — 简述：`path:行号``。与变更无关的类别标 **N/A**。  

---

## G1 并发控制
> 默认使用悲观锁，**一锁二判三更新**（锁定资源 → 基于最新状态再校验 → 再更新），乐观引入须谨慎；正逆向流程若多锁，**锁顺序须一致**以防死锁。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G1.1 | 在事务内或者并发场景下，先读后写无锁/无 `SELECT FOR UPDATE` | P0 |
| G1.2 | 已加锁或已 `loadWithLock`，但**未基于最新单据状态/金额等做二次校验**即 `update` | P0 |
| G1.3 | 高冲突场景用乐观锁却无限重试、无退避 | P2 |
| G1.4 | 不同业务流程对多资源加锁**顺序不一致** | P1 |

**正例**：

```java
// 并行查询带超时
List<T> r = futures.stream()
    .map(f -> f.orTimeout(3, TimeUnit.SECONDS))
    .map(CompletableFuture::join)
    .toList();
```

## G2 幂等拦截
> 幂等新老要兼容，字段约束需一致，异常场景防击穿。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G2.1 | 写接口/消息消费无幂等键（业务单号、消息 id、去重表） | P0 |
| G2.2 | 重试、定时任务、MQ 重投未防重复落库/重复扣款 | P0 |
| G2.3 | 幂等键与上游约定不一致（字段名、粒度） | P0 |

**正例**：

```java
// 消费前插入幂等记录或利用 DB 唯一约束，失败则视为已处理
if (!idempotency.tryAcquire("order-paid:" + e.getEventId())) {
    return;
}
walletService.credit(e.getUserId(), e.getAmount());
```

## G3 事务控制
> 悬挂告警要及时，严禁造成空提交，确保最终一致性。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G3.1 | 把分布式事务当成本地强一致使用（跨库读后写无补偿） | P1 |
| G3.2 | `@Transactional` 范围过大含外部I/O（RPC/MQ/文件）| P1 |

**反例**：

```java
@Transactional
public void create(Order o) {
    orderMapper.insert(o);
    paymentClient.charge(o.getId()); // 外部调用在事务内，易长事务、难回滚
}
```

## G4 SQL与索引
> 使用要求最简化，业务逻辑归代码，索引不含隐式转换；MyBatis 须 `#{}` 预编译，避免 `${}` 与字符串拼 SQL。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G4.1 | 复杂业务分支全堆在 SQL（多层子查询、存储过程拼业务） | P1 |
| G4.2 | WHERE 对索引列做函数/隐式类型转换导致全表扫 | P1 |
| G4.3 | 无分页的大列表查询、深分页无优化 | P1 |

**正例**：类型与列一致；大查询 `limit` + 排序键游标分页。

## G5 消息（MQ）
> 一考虑重复投递，二考虑丢失延迟，三考虑消息乱序。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G5.1 | 消费逻辑未幂等、未考虑乱序 | P0 |

**要点**：至少消费端幂等；顺序要求用分区键或业务版本号。

## G6 缓存
> 缓存击穿要兜底，过期设计去热点，存储容量需考虑。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G6.1 | 数据缓存无超时时间 | P0 |
| G6.2 | 缓存与 DB 双写无策略（先删缓存、延迟双删等） | P1 |

**反例**：

```java
@Cacheable(value = "user", unless = "#result == null") // 未设置 TTL，用户改密后长期脏读
public User getUser(String id) { ... }
```

## G7 调度任务
> 调度重叠要避免，捞取数据可配置，熔断能力需具备。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G7.1 | 无分布式锁/分片，多实例重复捞同批数据 | P1 |
| G7.2 | 无开关/熔断，故障时无法一键停任务 | P1 |

## G8 防御编程
> 强弱依赖要合理，单点异常可切换，线程隔离和限流；**线程池内 ThreadLocal 必须 `remove`**。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G8.1 | 仅处理 happy path，`catch` 吞异常或只打 log 不告警 | P1 |
| G8.2 | 核心链路强依赖非核心、无超时/降级 | P1 |
| G8.3 | I/O 流、连接、锁在 finally 或 try-with-resources 中释放 | P0 |
| G8.4 | 线程池 / 定时任务在应用关闭时 shutdown | P1 |
| G8.5 | **ThreadLocal** 在线程池场景：`finally` 未 `remove()`，或以 `set(null)` 代替 `remove()` | P0 |
| G8.6 | 使用 `Executors.newCachedThreadPool` / `newFixedThreadPool` 等**默认无界队列**线程池承载业务流量 | P0 |

**正例**：

```java
// ✅ try-with-resources 自动释放
try (InputStream is = new FileInputStream(file)) {
    process(is);
}
```

## G9 网络调用（RPC/HTTP）
> 交互三态要牢记，换号重发高风险，错误码映射要谨慎。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G9.1 | 外部调用（HTTP/RPC/DB/Redis）未区分：成功 / 找不到服务 / 请求超时 / 响应超时 | P0 |
| G9.2 | 外部调用（HTTP/RPC/DB/Redis）未设置超时 | P0 |
| G9.3 | 重试前未查询最新状态，或者未用幂等ID重试 | P0 |

**正例**：

```java
// ✅ 设置连接 + 读超时
RequestConfig cfg = RequestConfig.custom()
    .setConnectTimeout(1000)
    .setSocketTimeout(3000).build();
```

| G10.2 | 契约变更无版本/开关，强耦合上下游同发 | P1 |

## G11 开发自测
> CR 单测接口测，断言回归边界值，系统质量不降分。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G11.1 | 新逻辑无单测或单测无断言（只调不通） | P2 |
| G11.2 | 未覆盖边界：空、最大值、并发路径 | P2 |
| G11.3 | 入参空值 / 空集合无防御性校验 | P0 |
| G11.4 | 数值运算未考虑溢出、除零、精度丢失；**金额**使用 `float`/`double` 运算或 `==` 比较 | P0 |

## G14 国际化 / 多租户 / 时区（按需）
> 注意币种有无分，金额计算要克隆，外汇买卖分方向；**金额推荐** `BigDecimal`/业务 `Money` 等多币种类型，勿用浮点。入口设值不可变，数据都要租户值，隔离拆分是本质。切点时区和日历，三者不能缺其一，时间戳存UTC。

| ID | 扫描信号 | 等级 |
|----|----------|------|
| G14.1 | 金额用 `double`、未区分币种最小单位（如日元） | P0 |
| G14.2 | 多租户表查询无租户条件、租户 ID 来自用户可控参数 | P0 |
| G14.3 | 存本地时区字符串、跨区展示错误；应用 `Instant`/UTC 存库 | P1 |
| G14.4 | `SimpleDateFormat`/`DateTimeFormatter` 等解析或格式化**未显式指定时区**（国际化/跨区展示场景） | P1 |

**正例**：`tenantId` 从 `SecurityContext`/线程上下文注入，与登录租户一致。

## G15 可灰度
> 生产之前先灰度，分批操作控比例，关键业务须盯盘。

| ID | 检查项 | 等级 |
|---|--------|------|
| G15.1 | 数据库表结构变更向前兼容（加列可选、不删列、不改列类型） | P0 |
| G15.2 | 新旧接口共存期间，旧接口保持可用 | P1 |
| G15.3 | 不兼容逻辑是否通过开关来控制切换，或者添加白名单控制 | P1 |

**示例**：

```sql
-- ❌ 不兼容变更 — 旧代码部署期间直接报错
ALTER TABLE orders DROP COLUMN remark;
ALTER TABLE orders CHANGE status order_status INT;

-- ✅ 向前兼容 — 加列可选，旧代码不受影响
ALTER TABLE orders ADD COLUMN sub_type VARCHAR(32) DEFAULT NULL;
```

## G16 可监控
> 变更前后配校验，监控巡检覆盖全，防御规则勤演练；**禁止空 `catch`**，捕获后须日志/告警/补救之一。

| ID | 检查项 | 等级 |
|---|--------|------|
| G16.1 | 核心链路有关键指标埋点（成功率、耗时、调用量） | P0 |
| G16.2 | 异常路径有日志输出且包含可追溯上下文（traceId / bizId） | P1 |
| G16.3 | 日志级别正确：业务异常 WARN，系统异常 ERROR，正常流程 INFO | P1 |
| G16.4 | **空 `catch`**；仅 `printStackTrace()`；资金/关键路径捕获异常后仍继续执行且无记录 | P0 |

**示例**：

```java
// ❌ 异常被吞 — 线上出问题无法排查
try {
    process(order);
} catch (Exception e) {
    return Result.fail("处理失败");
}
```

## G17 可应急
> 发现变更先回滚，容量不够降和限，容灾切换来兜底。

| ID | 检查项 | 等级 |
|---|--------|------|
| G17.1 | 功能开关支持紧急关闭（不需重启 / 发布） | P0 |
| G17.2 | 有降级预案：核心链路异常时返回兜底结果或走备用通道 | P0 |
| G17.3 | 数据变更有回滚脚本或补偿方案 | P1 |
