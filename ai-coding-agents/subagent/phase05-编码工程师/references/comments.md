# 注释规约

## 1. Javadoc规范

### 1.1 类、属性、方法注释

**【强制】** 类、类属性、类方法的注释必须使用javadoc规范，使用`/**内容*/`格式，不得使用`//xxx`方式。

**说明**：
- 在IDE编辑窗口中，javadoc方式会提示相关注释
- 生成javadoc可以正确输出相应注释
- 工程调用方法时，不进入方法即可悬浮提示方法、参数、返回值的意义

**正例**：
```java
/**
 * 用户服务类，处理用户相关业务逻辑
 */
public class UserService {
    /**
     * 用户最大数量
     */
    private static final int MAX_USER_COUNT = 1000;

    /**
     * 根据ID获取用户信息
     *
     * @param id 用户ID
     * @return 用户信息，不存在则返回null
     */
    public User getUserById(Long id) {
        // ...
    }
}
```

### 1.2 抽象方法注释

**【强制】** 所有的抽象方法（包括接口中的方法）必须要用javadoc注释、除了返回值、参数、异常说明外，还必须指出该方法做什么事情，实现什么功能。

**说明**：对子类的实现要求，或者调用注意事项，请一并说明。

### 1.3 类创建信息

**【强制】** 所有的类都必须添加创建者信息和创建日期。

**说明**：在设置模板时，注意IDEA的@author为`${USER}`，而eclipse的@author为`${user}`，大小写有区别。日期的设置统一为`yyyy/MM/dd`的格式。

**正例**：
```java
/**
 * 订单服务实现类
 *
 * @author yangguanbao
 * @date 2016/10/31
 */
public class OrderServiceImpl implements OrderService {
}
```

## 2. 方法内部注释

### 2.1 单行注释

**【强制】** 方法内部单行注释，在被注释语句上方另起一行，使用`//`注释。

### 2.2 多行注释

**【强制】** 方法内部多行注释使用`/* */`注释，注意与代码对齐。

**正例**：
```java
public void processOrder(Order order) {
    // 校验订单状态
    if (!order.isValid()) {
        throw new IllegalStateException("订单状态无效");
    }

    /*
     * 以下代码处理订单支付逻辑
     * 1. 冻结用户余额
     * 2. 扣除订单金额
     * 3. 更新订单状态
     */
    freezeBalance(order.getUserId(), order.getAmount());
}
```

## 3. 枚举注释

**【强制】** 所有的枚举类型字段必须要有注释，说明每个数据项的用途。

**正例**：
```java
public enum OrderStatusEnum {
    /** 待支付 */
    WAIT_PAY(0),

    /** 已支付 */
    PAID(1),

    /** 已发货 */
    SHIPPED(2),

    /** 已完成 */
    COMPLETED(3),

    /** 已取消 */
    CANCELLED(4);

    private final int code;

    OrderStatusEnum(int code) {
        this.code = code;
    }
}
```

## 4. 条件注释

**【强制】** 在if()...else if()...else...的结构中，else后的相关条件注释紧挨着放在执行语句的第一行。

**正例**：
```java
if (a == 1) {
    System.out.println('一');
} else if (a == 2) {
    // 如果等于2
    System.out.println('二');
} else if (a == 3) {
    // 如果等于3
    System.out.println('三');
} else {
    // 如果是其他值
    System.out.println('其他');
}
```

## 5. 注释语言选择

### 5.1 中文注释

**【推荐】** 与其"半吊子"英文来注释，不如用中文注释把问题说清楚。专有名词、关键字，保持英文原文即可。

**反例**：
```java
// "TCP连接超时"解释成"传输控制协议连接超时"，理解反而费脑筋
```

### 5.2 注释同步

**【推荐】** 代码修改的同时，注释也要进行相应的修改，尤其是参数、返回值、异常、核心逻辑等的修改。

**说明**：代码与注释更新不同步，就像路网与导航软件更新不同步一样。

## 6. 代码清理

### 6.1 未使用代码

**【推荐】** 在类中，删除未使用的任何字段和方法；在方法中，删除未使用的任何参数声明与内部变量。

### 6.2 注释掉的代码

**【参考】** 谨慎注释掉代码。在上方详细说明，而不是简单地注释掉；如果无用，则删除。

**说明**：代码被注释掉有两种可能性：
1. 后续会恢复此段代码逻辑
2. 永久不用

如果是前者，没有备注信息，难以知晓注释动机；后者干脆删掉即可。

### 6.3 特殊注释标记

**【参考】** 特殊注释标记，请注明标记人与标记时间。注意及时处理这些标记。

1. **TODO**：表示需要实现，但目前还未实现的功能
   ```java
   // TODO: (张三, 2024/01/15, [预计2024/01/20完成]) 添加订单退款逻辑
   ```

2. **FIXME**：标记某代码是错误的，而且不能工作，需要及时纠正
   ```java
   // FIXME: (李四, 2024/01/15, [预计2024/01/16完成]) 并发场景下有线程安全问题
   ```

## 7. 注释质量

### 7.1 注释原则

**【参考】** 对于注释的要求：
1. 能够准确反映设计思想和代码逻辑
2. 能够描述业务含义，使别的程序员能够迅速了解到代码背后的信息

**说明**：完全没有注释的大段代码对于阅读者形同天书，注释是给自己看的，即使隔很长时间，也能清晰理解当时的思路；注释也是给继任者看的，使其能够快速接替自己的工作。

### 7.2 精简原则

**【参考】** 好的命名、代码结构是自解释的，注释力求精简准确、表达到位。

**说明**：避免出现注释的一个极端：过多过滥的注释。代码的逻辑一旦修改，修改注释又是相当大的负担。

**反例**：
```java
// put elephant into fridge
put(elephant, fridge);
```

**说明**：方法名put，加上两个有意义的变量名elephant和fridge，已经说明了这是在干什么，语义清晰的代码不需要额外的注释。
