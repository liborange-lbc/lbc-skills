# 编码实现执行指南

> coder 子Agent 在 Phase 5 使用。目标：按系分文档实现代码 + 单元测试。

## Input

| 文件 | 必须 | 说明 |
|------|------|------|
| `{需求目录}/需求清单.md` | 是 | 实现 checklist，逐条完成 |
| `{需求目录}/系分文档.md` | 是 | 实现依据，严格按设计实现 |
| `{需求目录}/需求追踪矩阵.md` | 是 | 需求→接口→数据变更的映射 |
| `{OUTPUT_DIR}/经验库.md` | 否 | 避免重犯已知问题（如存在则必读） |
| EXTENSION_DOCS.coding | 否 | 代码军规、编码规范等 |

## Execution

### Step 0: 阅读上下文

1. 读取需求清单，了解本批次要实现的需求条目
2. 读取系分文档中对应章节（通过追踪矩阵定位）
3. **必读经验库**（如存在），标记与本次实现相关的经验
4. 如有 EXTENSION_DOCS.coding（代码军规），通读一遍

### Step 1: 批量实现

按 BATCH_SIZE 分批，每批执行：

```
for each R{NN} in batch:
  1. 读取追踪矩阵中该需求对应的接口设计和数据变更
  2. 实现代码（严格按系分，不得擅自修改设计）
  3. 编写对应的单元测试（覆盖验收标准）
  4. 更新需求清单中该条目状态为 💻
```

### Step 2: 自检

每批完成后自检：

- [ ] 每条需求都有对应的代码实现
- [ ] 每条需求至少 1 个测试，测试覆盖验收标准
- [ ] 代码风格与项目现有代码一致
- [ ] 如有代码军规，逐条核对
- [ ] 无硬编码配置值
- [ ] 错误处理完整

## Output

| 产出 | 位置 | 说明 |
|------|------|------|
| 代码文件 | 项目代码目录 | 按项目结构组织 |
| 测试文件 | 项目测试目录 | 每条需求 ≥ 1 个测试 |
| 需求清单更新 | `{需求目录}/需求清单.md` | 已实现条目状态改为 💻 |
| 编码日志 | `{需求目录}/log/phase5-编码.md` | 记录实现详情 |

### 编码日志格式

```markdown
# 编码日志

## Batch 1: R01-R03

### R01 — {需求描述}
- 实现文件: src/api/register.ts, src/services/auth.ts
- 测试文件: tests/register.test.ts
- 测试数量: 3 个（正常注册、重复手机号、验证码过期）
- 状态: 💻 完成

### R02 — {需求描述}
- ...
```

### 文档质量要求

- [ ] 编码日志中每条需求都记录了实现文件和测试文件
- [ ] 需求清单中已实现的条目状态已更新
- [ ] 日志中无"待定"或"TODO"字样

## 行为约束

1. **严格按系分实现** — 不得擅自修改设计方案。有疑问写入日志，由主Agent转达
2. **每条需求 ≥ 1个测试** — 测试覆盖需求的验收标准
3. **增量追加** — 不重写已完成的代码
4. **匹配现有风格** — 命名、缩进、模式与项目现有代码一致
5. **扩展文档遵循** — 如有代码军规，每条规则作为编码约束

## 修正模式（Resume）

当主Agent Resume 本Agent修复CR问题时：

1. 读取 CR 报告中的具体修复指令
2. **定点修复** — 只改评审指出的问题，不顺手改其他地方
3. 修复完成后写入 `{需求目录}/log/phase6-迭代-{N}.md`
4. 更新经验库 — 如果修复揭示了可复用的模式
