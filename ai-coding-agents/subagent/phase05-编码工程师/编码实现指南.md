# 编码实现执行指南

> coder 子Agent 在 Phase 5 使用。目标：按系分文档逐模块实现代码 + 单元测试，遵循 5 阶段工作流（READ -> TEST -> IMPL -> CHECK -> DOCS）。

## Input

| 文件 | 必须 | 说明 |
|------|------|------|
| `{需求目录}/需求清单.md` | 是 | 实现 checklist，逐条完成 |
| `{需求目录}/系分文档.md` | 是 | 实现依据，严格按设计实现 |
| `{需求目录}/需求追踪矩阵.md` | 是 | 需求->接口->数据变更的映射 |
| `{OUTPUT_DIR}/经验库.md` | 否 | 避免重犯已知问题（如存在则必读） |
| EXTENSION_DOCS.coding | 否 | 代码军规、编码规范等 |

## Execution

> **重要**: P5 由主Agent编排，根据模块依赖关系决定串行或并行启动 coder 子Agent。  
> 每个 coder 子Agent 只负责自己分配到的模块，执行五阶段工作流。

### Step 0: 依赖分析与编排（主Agent执行）

**【强制】** 主Agent在启动 coder 之前，必须完成依赖分析：

1. **提取模块依赖图** — 从系分文档中提取：
   - 接口调用关系（Service A 调用 Service B）
   - 数据模型外键关系（表 A 引用表 B 的字段）
   - 公共代码依赖（多个模块共用的 config、util、基类）

2. **拓扑排序分层** — 按依赖方向分层：
   - **Layer -1（公共层）**: 被多个模块依赖的公共代码（config、枚举、基类、工具类）
   - **Layer 0**: 无依赖的基础模块
   - **Layer 1**: 依赖 Layer 0 的模块
   - **Layer N**: 依赖 Layer N-1 的模块
   - 同层内模块无互相依赖，可并行

3. **生成编排计划** — 写入 `{需求目录}/log/phase5-编排计划.md`：

```markdown
# P5 编码编排计划

## 模块依赖图
（Mermaid graph — 模块间依赖箭头）

## 分层编排

| 层级 | 模块 | 依赖 | 执行方式 | 分配Agent | 预期产出文件 |
|------|------|------|----------|----------|-------------|
| L-1 | common(config/util) | 无 | 串行(单coder先完成) | P5-编码工程师-common | src/common/*, src/config/* |
| L0 | catalog | 无 | 并行 | P5-编码工程师-catalog | src/*/catalog/* |
| L0 | config-mgmt | 无 | 并行 | P5-编码工程师-config-mgmt | src/*/configmgmt/* |
| L1 | metadata | catalog | 并行(L0完成后) | P5-编码工程师-metadata | src/*/metadata/* |
| L1 | auth | config-mgmt | 并行(L0完成后) | P5-编码工程师-auth | src/*/auth/* |
| L2 | identity | metadata, auth | 串行(L1完成后) | P5-编码工程师-identity | src/*/identity/* |

## 文件归属索引（修正循环定位用）

> 本表分两阶段填写：
> **Phase 1（coder 启动前）**: 主Agent 根据系分文档中的接口/数据模型，预填"预期产出文件"列（best-guess）
> **Phase 2（coder 完成后）**: 主Agent 从各 coder 编码日志中提取实际产出文件，回填"实际产出文件"列
> P6 CR FAIL 时，主Agent 通过"实际产出文件"列将 `文件:行号` 映射到对应 coder 的 agentId。

| 归属模块 | 归属Agent | 预期产出文件 | 实际产出文件（coder完成后回填） |
|---------|----------|-------------|---------------------------|
| catalog | P5-编码工程师-catalog | src/*/catalog/* | （回填） |

### 公共代码修复归属规则

公共代码（L-1 层）的修复归属：
- **公共代码文件**（如 ErrorCode.java、BaseService.java）的 bug → **一律 Resume L-1 层 coder**
- **模块代码调用公共代码不当** → Resume 该模块的 coder（问题在调用方，不在公共代码）
- 判断依据：CR 报告中 `文件:行号` 所在文件属于哪一层

## 公共代码清单
- src/common/ErrorCode.java — 被 catalog, metadata, identity 依赖
- src/config/AppConfig.java — 被所有模块依赖

## 风险项
- {如有循环依赖或无法拆分的情况，在此说明}
```

4. **按层级启动 coder** — 主Agent 按编排计划执行：
   - Layer -1: 启动单个 coder 完成公共代码，等待完成
   - Layer 0: **并行启动**多个 coder（`Agent(run_in_background: true)`）
   - 等待 Layer 0 全部完成
   - Layer 1: **并行启动**多个 coder
   - 依此类推直到所有层级完成

5. **循环依赖检测** — 拓扑排序时如发现循环依赖：
   - 在编排计划风险项中标记为"架构级问题"
   - 主Agent 向用户报告："系分设计中存在循环依赖（A↔B），建议返回 Phase 3 重新设计"
   - 用户确认继续 → 将循环依赖的模块合并到同一层级，降级为串行由单 coder 处理
   - 用户要求修复 → 回退到 Phase 3

6. **单模块降级** — 如果只有 1-2 个模块，或所有模块强依赖串行，则退回单 coder 模式

### Step 0.5: 创建模块进度追踪表（每个 coder 维护自己的模块）

每个 coder 子Agent 启动后，为分配到的模块创建进度追踪：

```markdown
## 模块进度追踪

| 序号 | 模块 | 层级 | READ | TEST | IMPL | CHECK | DOCS | 状态 |
|:----:|------|:----:|:----:|:----:|:----:|:-----:|:----:|------|
| 1 | catalog | L0 | ⬜ | ⬜ | ⬜ | ⬜ | ⬜ | 待开始 |
```

状态取值：待开始 -> 进行中 -> 已完成

### Step 1: 五阶段工作流（每个 coder 子Agent 执行）

**【强制】** 每个模块必须按顺序完成以下 5 个阶段，缺一不可。代码必须语法正确，条件允许时执行编译和单测验证。

---

#### 阶段一：READ（读取上下文）

**输入**：系分方案

**动作**：
1. 读取根目录的 SSOT.md（若存在）获取：架构文档路径、模块文档路径、项目规范位置
2. 读取需求清单，了解本批次要实现的需求条目
3. 读取系分文档中对应章节（通过追踪矩阵定位）
4. **必读经验库**（如存在），标记与本次实现相关的经验
5. 如有 EXTENSION_DOCS.coding（代码军规），通读一遍
6. **按需加载** references 规范文档（见下方规范索引表）

**输出**：

```markdown
### READ: {module_name}

**模块职责**：{一句话描述}

**关键类列表**：
- {Entity}DO - 数据对象
- {Entity}VO - 视图对象
- {Entity}Mapper - 数据访问
- {Entity}Service - 业务服务
- {Entity}Controller - 控制器

**依赖关系**：{依赖的其他模块或服务}

**已加载规范**：
- [x] collections.md
- [x] concurrency.md
- [x] references/antdigital/antdigital-gateway.md
```

更新进度表：READ 列标记为 ✅，状态改为 进行中

---

#### 阶段二：TEST（生成单测）

**【强制】** 在实现业务代码之前，必须先生成单元测试（TDD）。

**动作**：
1. 确定被测类（SUT）：Service 实现类
2. 读取 references 中的单测相关规范
3. 生成测试类 `src/test/java/.../{ClassName}Test.java`
4. 遵循 AAA 模式（Arrange-Act-Assert）

**输出**：

```markdown
### TEST: {module_name}

**测试文件**：`{package}/{ClassName}Test.java`

**测试方法列表**：
| 方法 | 测试场景 | 状态 |
|------|----------|:----:|
| should_returnList_when_validRequest | 正常路径 | ✅ |
| should_throwException_when_paramNull | 参数校验 | ✅ |
| should_throwException_when_dataNotFound | 数据不存在 | ✅ |

**测试覆盖摘要**：
- 被测类: {ClassName}
- 测试方法数: {N}
- 覆盖场景: 正常路径 ✓, 参数校验 ✓, 异常处理 ✓, 边界值 ✓
```

更新进度表：TEST 列标记为 ✅

---

#### 阶段三：IMPL（实现代码）

**动作**：
1. 按 Entity -> Mapper -> Service -> Controller 顺序实现业务代码
2. 确保代码语法正确、依赖存在
3. 条件允许时运行 `mvn compile -DskipTests` 验证编译通过
4. 确保符合命名、异常、安全等规范
5. 严格按系分实现，不得擅自修改设计

**输出**：

```markdown
### IMPL: {module_name}

**已实现文件**：
- `model/entity/{Entity}DO.java`
- `model/dto/{Entity}VO.java`
- `model/dto/{Entity}Request.java`
- `dao/mapper/{Entity}Mapper.java`
- `resources/mapper/{Entity}Mapper.xml`
- `service/{Entity}Service.java`
- `service/impl/{Entity}ServiceImpl.java`
- `controller/{Entity}Controller.java`

**编译验证**：✅ 通过 / ⚠️ 环境受限 / ⏭️ 已跳过（原因：...）
```

更新进度表：IMPL 列标记为 ✅

---

#### 阶段四：CHECK（规范检查）

**【强制】** 输出规范检查清单。采用 L1 静态检查（必做） + L2 动态验证（条件执行）分层策略。

**L1 静态检查**：逐项检查规范符合情况。

**L2 动态验证**：
- 探测环境：`mvn -version`
- 成功 -> 执行 `mvn compile` 和 `mvn test`
- 失败 -> 标记跳过原因，输出待人工验证命令

**输出模板**：

```markdown
### CHECK: {module_name}

#### L1 静态检查

| 检查项 | 规范要求 | 符合情况 |
|--------|----------|:--------:|
| 命名规范 | 类名大驼峰、方法名小驼峰、常量全大写 | ✅ |
| 数科网关 | API字段 snake_case、响应结构正确 | ✅ |
| 异常日志 | 日志工具类/SLF4J + 占位符、自定义异常 | ✅ |
| 安全规范 | SQL参数化 #{ }、输入校验 @Valid | ✅ |
| MySQL规范 | 表名小写、必备字段、resultMap | ✅ |
| 单元测试 | 测试类存在、覆盖率摘要已输出 | ✅ |
| 常量定义 | 无魔法值、Long用大写L、常量分类 | ✅ |
| 集合处理 | toMap有mergeFunction、不在foreach中remove | ✅ |
| 并发处理 | ThreadPoolExecutor创建、ThreadLocal清理 | ✅ |
| 控制流 | switch有default、大括号完整、嵌套<=3层 | ✅ |
| 日期时间 | yyyy小写、使用JDK8 API、无sql.Date | ✅ |
| 注释规范 | Javadoc格式、枚举有注释、创建者信息 | ✅ |

#### L2 动态验证

| 验证项 | 状态 | 说明 |
|--------|:----:|------|
| 编译验证 | ✅/⚠️ | {通过 / 跳过原因} |
| 单测验证 | ✅/⚠️ | {通过 / 跳过原因} |

#### 待人工验证

以下命令请在本地执行，确认代码质量：

mvn compile -DskipTests
mvn test -Dtest={ClassName}Test

**发现问题**：{无 或 列出问题并修复}
```

更新进度表：CHECK 列标记为 ✅

## 行为约束

1. **严格按系分实现** -- 不得擅自修改设计方案。有疑问写入日志，由主Agent转达
2. **TDD 先测试后实现** -- 每个模块先生成单测（阶段二），再实现代码（阶段三）
3. **五阶段不可跳跃** -- READ -> TEST -> IMPL -> CHECK -> DOCS 必须按序完成
4. **按需加载规范** -- 在 READ 阶段根据模块特点加载对应规范，不要一次性读完
5. **增量追加** -- 不重写已完成的代码
6. **匹配现有风格** -- 命名、缩进、模式与项目现有代码一致
7. **扩展文档遵循** -- 如有代码军规，每条规则作为编码约束
8. **L1+L2 双层检查** -- CHECK 阶段必须输出 L1 静态检查表 + L2 动态验证结果

## 修正模式（Resume）

当主Agent Resume coder 修复CR/测试问题时，仍须保障 TDD 闭环。

### 并行模式下的 coder 定位（主Agent执行）

```
# Step 1: 从 CR 报告提取问题文件
Grep: log/phase6-CR-*.md → "文件:行号" 列

# Step 2: 查编排计划的文件归属索引，定位归属模块和Agent
Grep: log/phase5-编排计划.md → "文件归属索引" 表 → 找到对应 Agent 名称

# Step 3: 从执行记录查 agentId
Grep: log/执行日志.md → "{Agent名称}" → 提取 agentId

# Step 4: Resume 对应 coder
SendMessage(to: "{agentId}", message: "修复以下问题: ...")
```

如果问题涉及多个模块的 coder，需分别 Resume 各自的 coder。  
如果问题涉及公共代码（L-1 层），Resume 公共层 coder。

### 每个 coder 的修复流程

1. 读取 CR/测试报告中的具体修复指令
2. **TEST — 先补/改测试**：
   - 针对每个修复点，先更新或新增对应的单元测试，使其覆盖修复场景
   - 新增的测试此时应为 RED（因为代码尚未修复）
3. **IMPL — 定点修复代码**：
   - 只改评审/测试指出的问题，不顺手改其他地方
   - 修复后测试应变 GREEN
4. **CHECK — 对修复部分重新执行规范检查**：
   - L1 静态检查：仅检查修改涉及的文件
   - L2 动态验证：条件允许时运行 `mvn test -Dtest={修改的TestClass}`
5. 修复完成后写入 `{需求目录}/log/phase6-迭代-{N}.md`（或 `phase7-迭代-{N}.md`）
6. 更新经验库 — 如果修复揭示了可复用的模式
