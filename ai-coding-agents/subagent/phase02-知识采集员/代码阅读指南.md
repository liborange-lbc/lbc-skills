# 知识采集执行指南

> collector 子Agent 在 Phase 2 使用。目标：建立设计所需的全部上下文。

## Input

| 文件 | 必须 | 说明 |
|------|------|------|
| `{需求目录}/需求清单.md` | 是 | 确定代码阅读范围 |
| EXTENSION_DOCS.requirement | 否 | 补充业务上下文（直接读取，作为背景知识） |
| 项目代码库 | 是 | 代码阅读对象 |

## Execution

### Step 0: 确定代码阅读范围

1. 读取需求清单，提取涉及的功能模块
2. **只读与需求相关的模块**，不做全库扫描
3. 如需求涉及多个模块，按依赖顺序逐模块阅读

### Step 1: GitNexus 索引

```bash
# 索引项目（首次执行，后续增量更新）
gitnexus analyze {PROJECT_ROOT} --name {project-name}

# 验证索引状态
gitnexus status
```

如果项目已有索引（`gitnexus list` 可查），跳过此步。  
**降级方案**: 如果 GitNexus 索引失败（非 git 仓库、依赖缺失等），跳过 GitNexus 相关步骤，仅使用六步下钻法。

### Step 2: 快速全局视图（GitNexus）

```bash
# 用自然语言搜索相关代码流程
gitnexus query "{需求关键词}" -r {project-name} --goal "找到相关的全部代码路径"

# 查看核心符号的 360 度上下文
gitnexus context {CoreSymbol} -r {project-name} --content
```

### Step 3: 六步下钻法（深入关键模块）

| Step | 目标 | 查找 | 记录 |
|------|------|------|------|
| 1 模块入口 | Controller/Router/Handler | URL、请求响应类型、鉴权 | 入口清单 |
| 2 领域模型 | Entity/Model/Schema | 字段、关联、枚举、约束 | 数据模型图 |
| 3 核心业务 | Service/UseCase 实现 | 事务边界、锁、校验、外部调用 | 调用链 + 业务规则 |
| 4 数据访问 | Repository/DAO/Mapper | 复杂查询、索引、批量操作 | 查询模式 + 性能风险 |
| 5 边界交互 | 外部服务/MQ/缓存/定时任务 | 失败处理、重试、幂等、超时 | 依赖图 + 故障模式 |
| 6 配置基础设施 | 配置文件/迁移脚本/CI | 环境差异、功能开关 | 配置清单 |

### Step 4: 知识库查阅

按优先级读取外部文档，详见 [知识查阅规则.md](知识查阅规则.md)。

### Step 5: 停止条件

满足以下**全部**条件时停止采集：
- 需求清单中**每条需求涉及的模块**都已阅读
- **核心调用链**已理清（入口→业务逻辑→数据访问→外部依赖）
- **风险点**已识别并记录
- **现有约束**（幂等、事务、缓存等）已梳理

## Output

产出**两个文件**，写入 `{需求目录}/`：

### 产出 1: 代码阅读报告.md

```markdown
# 代码阅读报告

## 模块概览
- 涉及模块: {模块列表}
- 技术栈: {语言/框架/数据库}
- GitNexus索引: {project-name} ({N}文件, {N}符号, {N}边)

## 入口清单
| 接口 | 方法 | 路径 | 鉴权 |
|------|------|------|------|

## 数据模型
{ER图或表格，标注表名、关键字段、关联关系}

## 核心调用链
{Mermaid时序图或文字描述，标注事务边界}

## 依赖关系
{外部服务/中间件依赖图，标注超时和降级策略}

## 风险标记
| # | 风险 | 位置 | 影响 | 建议 |
|---|------|------|------|------|

## 现有约束
{已有的幂等/事务/缓存/锁机制，designer 必须遵循}
```

### 产出 2: 知识摘要.md

> designer 在 Phase 3 读取此文件作为设计背景。要求：**简洁、结论导向、可直接用于设计决策**。

```markdown
# 知识摘要

## 技术栈
- 语言: {language}
- 框架: {framework}
- 数据库: {database}
- 中间件: {Redis/MQ/ES 等}

## 涉及模块概要
| 模块 | 职责 | 关键类/文件 | 与本需求的关系 |
|------|------|-----------|--------------|

## 现有接口（与本需求相关）
| 接口 | 方法 | 路径 | 说明 | 可复用？ |
|------|------|------|------|---------|

## 数据模型现状
| 表/Collection | 关键字段 | 与本需求的关系 | 需要变更？ |
|--------------|---------|--------------|----------|

## 业务规则（从代码/文档中提取）
1. {规则1} — [来源: {文件:行号 或 文档名}]
2. {规则2} — [来源: ...]

## 技术约束（设计时必须遵循）
1. {约束1，如"该模块使用乐观锁，不支持悲观锁"}
2. {约束2，如"外部支付接口超时 3s，必须有降级"}

## 风险与建议
| 风险 | 影响 | 建议 |
|------|------|------|

## 扩展文档摘要（来自 EXTENSION_DOCS）
{如果有扩展文档，提取与本需求相关的关键信息，标注来源}
```

### 文档质量要求

- [ ] 所有引用标注来源（`[来源: 文件:行号]` 或 `[来源: 文档名, 章节]`）
- [ ] 不含编造内容（查不到写"未找到"）
- [ ] 知识摘要中每条技术约束都有代码证据
- [ ] 风险标记至少覆盖：性能瓶颈、数据一致性、外部依赖故障
- [ ] 两份文档都已写入 `{需求目录}/`
