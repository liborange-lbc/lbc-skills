# 架构约束

> designer 子Agent 在 Phase 3 系分编写时必须遵循。reviewer 在 Phase 4 逐条核对。

## MUST — 缺一项评审必不过

| # | 约束 | 为什么 |
|---|------|--------|
| 1 | 所有写接口必须说明幂等机制 | 网络重试、MQ重复消费导致重复写入 |
| 2 | 新接口字段只增不删不改类型 | 破坏向前兼容导致已上线调用方报错 |
| 3 | 外部依赖必须有降级/熔断策略 | 下游故障级联拖垮上游 |
| 4 | 新接口必须说明鉴权方案 | 裸露接口 = 安全事故 |
| 5 | 事务范围尽可能小 | 大事务锁表、拖慢响应、阻塞连接池 |
| 6 | 关键路径必须有日志/指标/告警 | 线上不可观测 = 无法排查 |
| 7 | 部署架构不可出现单点问题 | 重要业务需同城双机房、高可用部署 |
| 8 | 数据模型变更必须向前兼容 | 新增属性需有默认值或非必填，不可删除属性、不可收紧类型 |
| 9 | 接口变更必须向前兼容 | 上游不升级时依然能正确调用变更后接口 |
| 10 | 变更三板斧（可监控/可灰度/可应急）必须设计 | 确保变更可控、可回退 |

## MUST NOT — 出现直接 CRITICAL

| # | 禁止 | 后果 |
|---|------|------|
| 1 | 事务内调用外部服务 (RPC/HTTP/MQ) | 连接池耗尽 |
| 2 | 单事务涉及 >3 张表且无理由 | 死锁概率剧增 |
| 3 | SELECT * 或无 LIMIT 查询 | 全表扫描或OOM |
| 4 | 硬编码配置值 | 环境切换必出问题 |
| 5 | 跨模块直接访问对方数据库 | 模块耦合无法演进 |
| 6 | 模块间出现循环依赖 | 架构腐化、无法独立演进 |
| 7 | 接口暴露敏感信息且不脱敏 | 用户手机号/密码/证件号码泄露 |

## 数据库设计约束

数据库设计须遵循 [references/db.md](references/db.md) 规范，关键强制项：

- 表名、字段名全部小写，由英文单词与下划线组成
- 新增表必须有整形单列主键，禁止联合主键
- 新增表模板需有 id, gmt_create, gmt_modified 三个字段
- 已上线表禁止删除、禁止删除字段、禁止将允许为空改为非空
- 新增字段需设置 default 值或不能强制 not null
- 禁止使用 timestamp（用 datetime）、float/double（用 decimal）、enum 类型
- 禁止使用外键、存储过程、视图、触发器

## 技术栈适配

以上约束源自 Java/微服务场景。非 Java 项目做语义映射：

- **幂等、鉴权、可观测性** — 永远适用，语言无关
- 事务范围 → ORM session/transaction 范围
- SELECT * → ORM 默认 select all / lazy load
- 跨模块数据库 → 跨 package 直接 import model
