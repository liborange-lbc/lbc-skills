数据库设计规范

# 关于命名
-【强制】新增库名、表名限制命名长度，建议表名及字段名字符总长度小于26
-【强制】表名、字段名全部小写，由英文单词与下划线组成
-【强制】禁止用保留字（关键词）做数据库名、表名、字段名、索引名，参考官方手册
-【强制】新增表名、字段名建议以字母开头，并采用字母、数字、下划线的一种或者多种组合
-【强制】主键索引应以pk_开头，唯一索引要以uk_开头，普通索引要以idx_打头
-【强制】全链路中用到的影子表以__test开头(双下划线开头)
-【推荐】库名最好和应用名保持一致
-【推荐】所属同一模块的表建议以相同前缀命名
-【推荐】分库，分表 均以[库名 | 表名]_****，后面的数字位数一致，建议统一采用 2,4,6 位数字补齐
-【推荐】布尔意义的字段以is_作为前缀，后接动词过去分词。如is_deleted
-【推荐】多表中的相同列，必须保证列名一致，数据类型一致
-【推荐】表名不使用复数名词
-【推荐】表的命名最好是遵循“业务名称_表的作用”，如：alipay_task / force_project / trade_config

# 关于结构设计
-【强制】新增表必须要有整形的单列主键，
-【强制】新增列必须在已有字段后面
-【强制】禁止使用外键、存储过程、存储函数、视图、触发器等非标准化语法
-【强制】库，表 引擎建议默认为InnoDB，事务隔离级别为ReadCommited，禁止指定表存储引擎
-【强制】多表中的相同列，必须保证列定义一致
-【强制】表主键不可使用联合主键(分区表除外，可加上分区键作为主键)，分库分表的表必须有全局的主键
-【强制】已上线的表禁止删除
-【强制】已上线表禁止将允许为空的列修改成非空列，否则锁表
-【强制】已上线表禁止删除字段
-【强制】已上线表新增字段需要设置default值，或者不能强制not null
-【强制】分库分表的字段名，字段类型和字段顺序要相同
-【强制】表被索引列必须定义为not null，并设置default值
-【强制】禁止使用varchar类型作为主键语句设计
-【强制】索引根据左前缀原则，当建立一个联合索引(a,b,c)，则查询条件里面只有包含(a)或(a,b)或(a,b,c)的时候才能走索引,(a,c)作为条件的时候只能使用到a列索引,所以这个时候要确定a的返回列一定不能太多，不然语句设计就不合理,(b,c)则不能走索引
-【推荐】数据库编码最好设置为utf8或者utf8mb4
-【推荐】新增表或字段不能指定字符集，建议 库，表，字段 采用相同字符集
-【推荐】新增表模板自动有 id,gmt_create,gmt_modified 3个字段，并对时间字段在代码逻辑中进行维护，便于数据分析及历史数据清理动作
-【推荐】新增表或字段建议有注释
-【推荐】建议采用BTREE索引，避免使用BITMAP索引 和 HASH索引
-【推荐】如果修改字段含义或对字段表示的状态追加时，需要及时更新字段注释
-【推荐】字段允许适当冗余，以提高查询性能，但必须考虑数据一致。冗余字段应遵循：不是频繁修改的字段；不是 varchar 超长字段，更不能是 text 字段；不是唯一索引的字段
-【推荐】索引不是越多越好，越多的索引带来的就是更高的索引维护成本，包含CPU计算消耗，索引建立时增加的IO开销等，因此一定要合理建立索引
-【推荐】联合索引应该选择筛选性更优的列值放在最前面，比如单号、userid等，type，status等筛选性一般的不建议放在最前面
-【推荐】如果存储的字符串长度几乎相等，使用 char 定长字符串类型
-【推荐】表的设计须遵循第一范式（即不允许字段出现二义性；），尽量达到第二范式及第三范式(从实际情况出发，取平衡点。当有设计不遵循第一范式的情况，请及时反馈并进行确认)
-【推荐】每个表的字段数目不要超过100个
-【推荐】单表一到两年内数据量超过500w或数据容量超过10G考虑分表，且需要提前考虑历史数据迁移或应用自行删除历史数据
-【推荐】日志类数据不建议存储在MySQL上，优先考虑Hbase或OB，如需要存储请找DBA评估使用压缩表存储
-【推荐】可变长度设计(例如varchar)一定要按照需要设计，同时控制单行长度，如果过大会降低数据库Buffer命中率，导致更新、查询性能下降

# 关于类型
-【强制】禁止使用timestamp类型，使用datetime类型并指定默认值，timestamp 存在2038存储限制
-【强制】禁止使用float或double，全部使用decimal类型代替
-【强制】禁止使用enum（枚举）类型
-【推荐】有时间精度要求的业务，可以使用datetime(6); 对精度没要求的，设置datetime
