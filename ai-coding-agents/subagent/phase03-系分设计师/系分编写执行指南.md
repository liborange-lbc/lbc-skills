# 系分编写执行指南

> designer 子Agent 在 Phase 3 使用。目标：按 9 步设计流程生成结构化系分文档 + 需求追踪矩阵。

## 核心原则

- **分步确认** — 每个步骤处理完成后，先进行确认再进入后续步骤
- **多给方案建议** — 对需要设计选型的场景，提供多个选项或建议让用户选择，而不是只提问题
- **不确定的要先确认** — 不确定的内容必须询问用户，宁可花更多时间互动，也不要生成不准确的内容
- **尊重事实** — 不要造需求，所有需求功能点都要能从需求文档中进行追溯
- **不遗漏** — 所有需求中内容都要有体现，不能缺失
- **先澄清再细化** — 范围未定时不展开过多实现细节
- **一次聚焦** — 若需求很大，先做最小可行范围的设计，再拆迭代
- **可追溯** — 设计中的假设、待确认项单独列出，避免遗漏

## Input

| 文件 | 必须 | 说明 |
|------|------|------|
| `{需求目录}/需求清单.md` | 是 | 需求条目列表，每条需求必须有对应设计 |
| `{需求目录}/代码阅读报告.md` | 是 | 现有代码结构、约束、风险 |
| `{需求目录}/知识摘要.md` | 是 | 技术约束和业务规则（设计决策依据） |
| EXTENSION_DOCS.design | 否 | 业务知识文档、数据字典等 |
| DESIGN_TEMPLATE_URL | 否 | 外部系分模板（空则用内置模板） |

## Execution — 9 步设计流程

每步结束先展示该步骤的结果并进行确认审查，确认无误后再进入下一步。

```
进度追踪：
- [ ] 1. 需求与范围澄清
- [ ] 2. 架构与模块划分
- [ ] 3. 数据模型与存储
- [ ] 4. 接口设计（oneapi/对外/内部）
- [ ] 5. 功能模块实现流程
- [ ] 6. 非功能性需求设计
- [ ] 7. 变更三板斧设计
- [ ] 8. 产出设计文档
- [ ] 9. 方案检查
```

### Step 1: 需求与范围澄清

1. 读取需求清单，逐条理解每条需求的描述和验收标准
2. 读取知识摘要，掌握技术约束和现有模块结构
3. 读取代码阅读报告，了解现有接口、数据模型、调用链
4. 如有 EXTENSION_DOCS.design，读取业务知识文档
5. 从需求中提炼：**做什么、给谁用、边界在哪**
6. 明确：核心功能列表、非功能要求（性能/可用性/安全）、不在本期范围的内容
7. 若有歧义，列出假设与待确认项

**产出**：一段「需求与范围」摘要（含核心功能、约束、排除范围）

### Step 2: 架构与模块划分 → 骨架确认

> 本步骤注重总体面上的把握和思考，不要过早深入实现细节

1. 检查项目内的架构文档（`docs/ARCHITECTURE.md`）是否有历史架构信息，如果有，基于该架构进行调整完善
2. 根据范围确定：单体/多服务、与现有系统的边界
3. 划分模块/子域：每个模块职责单一、接口清晰；功能架构在模块下进一步细分到功能点
4. 若涉及外部系统，标出集成点与调用关系，标识协议、接口类型，形成集成架构（包括集成架构图及说明）
5. 系统部署架构设计：
   - 公有云：重要业务应用必须考虑高可用，同城双机房，云原生部署方案，不可出现单点问题
   - 私有化：确认是容器化还是虚拟机部署
   - 未说明公有云还是私有化：需先确认再做设计
6. **不可出现循环依赖**
7. 按设计文档模板（参见 `设计文档模板.md`）生成系分文档骨架（仅章节标题）
8. 将骨架写入 `{需求目录}/系分文档.md`，然后**结束本轮执行，返回主Agent**
9. 主Agent 将骨架转交用户确认（designer 不直接与用户对话）
10. 用户确认后，主Agent Resume designer 继续 Step 3

**产出**：功能架构图（Mermaid）+ 模块清单（名称、职责、依赖）；应用集成架构图（Mermaid）+ 调用集成关系说明；部署架构图（Mermaid）

### Step 3: 数据模型与存储

> 数据库模型设计需遵循规范 [references/db.md](references/db.md)

1. 先总体设计所有模型列表，再针对单个模型结合业务需求详细设计其具体结构
2. **领域模型**：核心实体、关键属性、实体间关系（ER 图）
3. **持久化**：表/集合设计、主键与关键索引
4. 若有缓存、MQ 等，说明用途与数据形态
5. **版本兼容性设计**：新版本升级时数据模型变更后在旧版本上依然能兼容——不能做属性删除、属性类型收紧，新增属性需有默认值或不能是必须的
6. 所有表必须详细定义，包括所有字段，每一行一个字段，不要合并

**产出**：核心实体关系图（Mermaid ER 图）、各表结构说明（字段、类型、约束、注释）

### Step 4: 接口设计

> 先总体梳理接口列表，再根据需求文档详细设计每个接口

1. **oneapi**：REST 路径、方法、出入参设计、示例、错误码约定
2. **对外 API**：REST 路径、方法、出入参设计、示例、错误码约定
3. **内部接口**：Service 间、与 integration 的调用约定（方法签名和详细出入参）
4. **依赖服务接口**：对于集成架构中存在交互关系的非本系统应用，需针对每个设计对应接口，不清晰的需要询问
5. **接口兼容性设计**：依赖相关接口的应用回滚到旧版本或不升级时，依然能完成正确调用
6. 每个接口标注关联需求ID、幂等机制、鉴权方案

**产出**：接口列表（名称、路径/方法、简要说明）+ 每个接口的详细说明（入参、出参、参数说明）

### Step 5: 功能模块设计

> 结合功能架构，针对每个模块下每个功能点进行深入分析，追求深度而非广度

1. 对具体功能模块设计时，先参考历史模块文档（`docs/modules/`），存在相应模块则基于已有模块修改
2. 修改如存在不兼容性，必须明确标识出来
3. 各子功能的详细实现：调用流程时序图、调用说明
4. 关键设计：核心算法（如有）、安全、性能、类图、状态图（如有）
5. 过程中需求不明确的，及时沟通

**产出**：功能模块及下属功能的调用流程时序图（Mermaid sequenceDiagram）、调用说明、关键点说明

### Step 6: 非功能性需求设计（如有）

1. **稳定性设计**：从部署角度、系统角度思考，突发流量下系统依然高可用
2. **高可用设计**：从部署、系统等维度思考，特别是第三方服务异常时如何降级
3. **安全性设计**：网络安全、垂直权限、水平越权、SQL注入等
4. **性能设计**：关键核心服务考虑本地缓存/Redis缓存，避免缓存击穿、缓存雪崩
5. **扩展性设计**：横向扩展、容量扩展，功能层面插件式设计（如今天依赖OSS明天可能换minio，上层无感）

**产出**：非功能性设计的详细举措及关键内容

### Step 7: 变更三板斧

1. **可监控**：变更内容需能从服务性能、结果、异常等角度监控。日志埋点至少包括：
   - 自身服务：调用的服务、处理结果、处理耗时
   - 依赖三方服务：调用的服务名称、处理结果、处理耗时
2. **可灰度**：除部署角度的灰度部署外，可考虑软件角度灰度引流（如按租户尾号灰度）
3. **可应急**：出问题时能通过开关控制切换到旧逻辑，或通过发布包回滚。回滚时需重点关注：
   - 下游服务是否做了不兼容变更（已有接口做了不兼容升级或删除了已有接口）
   - 自身对外接口也需考虑兼容性升级

**产出**：变更三板斧的关键思考与设计

### Step 8: 产出设计文档 + 需求追踪矩阵

1. 将 Step 1-7 内容整理为完整系分文档，格式见 [设计文档模板.md](设计文档模板.md)
2. 写入 `{需求目录}/系分文档.md`

3. 生成**独立的**需求追踪矩阵文件：

```markdown
# 需求追踪矩阵

| 需求ID | 需求描述 | 设计章节 | 接口 | 数据变更 | 状态 |
|--------|---------|---------|------|---------|------|
| R01 | 手机号注册 | §4.1 | POST /api/v1/register | t_user 新增 | ✅ 已覆盖 |
| R02 | 唯一性校验 | §4.1 | 同上 | idx_phone UNIQUE | ✅ 已覆盖 |
```

**每条需求必须有映射**。如果某条需求无法映射到具体设计章节，标记为"❌ 未覆盖"并说明原因。

### Step 9: 方案检查

完成后逐项自检：

1. **架构模块**：是否高内聚低耦合
2. **集成架构**：依赖关系是否合理，是否有不必要的依赖，下游出问题时本系统能否保证服务可用
3. **部署架构**：是否有单点问题
4. **数据库范式**：是否满足1NF（必须），未提升到更高范式的原因是否合理
5. **接口敏感信息**：是否存在用户手机号、密码、邮箱、证件号码等，是否需要脱敏
6. **接口兼容性**：修改的接口，上游不升级时是否依然能正常调用
7. **功能模块流程**：是否清晰，是否存在并发风险，有无更高效方案（多方案需列优缺点及推荐理由）
8. **定时任务**（如有）：是否单点，能否横向扩容提升效率，是否存在并发问题
9. **非功能性**：是否考虑了稳定性、高可用、性能、安全、扩展性
10. **变更三板斧**：
    - 监控埋点是否完整
    - 灰度方案可行性分析、多方案比较及推荐建议
    - 可应急是否从下游问题、自身问题、上游问题等角度考虑
11. **架构约束核对**：逐条核对 [架构约束.md](架构约束.md) 中所有 MUST 约束 + 所有 MUST NOT 约束
12. **需求追踪矩阵覆盖率** = 100%
13. 所有引用标注来源

## Output

完成系分文档后，**更新需求清单状态**：将 `{需求目录}/需求清单.md` 中所有已覆盖需求的状态改为 📐。

产出**两个独立文件**，写入 `{需求目录}/`：

| 文件 | 说明 |
|------|------|
| `系分文档.md` | 完整系分文档，格式见 [设计文档模板.md](设计文档模板.md) |
| `需求追踪矩阵.md` | 独立文件，需求→设计章节→接口→数据变更的映射表 |

### 文档质量要求

- [ ] 系分文档包含所有骨架章节，无空章节
- [ ] 每个接口有完整字段：Path、Auth、Idempotent、关联需求、Request、Response、Error Codes
- [ ] 每个数据变更有完整字段：变更类型、字段表、索引、迁移、容量
- [ ] 数据库设计遵循 [references/db.md](references/db.md) 规范
- [ ] 至少一张核心流程时序图（Mermaid sequenceDiagram）
- [ ] 需求追踪矩阵覆盖率 = 100%（所有需求都有映射）
- [ ] 非功能设计覆盖稳定性/高可用/安全/性能/扩展性
- [ ] 变更三板斧（可监控/可灰度/可应急）完整

### 日志

写入 `{需求目录}/log/phase3-系分编写.md`

## 行为约束

1. **严格按需求设计** — 不造需求，所有功能点可追溯到需求文档
2. **分步确认** — 每步结果确认后才进入下一步
3. **版本兼容性** — 接口和数据模型变更必须考虑向前兼容
4. **多方案建议** — 有多个方案时列出优缺点和推荐理由
5. **架构约束遵循** — 严格遵循 [架构约束.md](架构约束.md) 中的 MUST / MUST NOT

## 修正模式（Resume）

当主Agent Resume 本Agent修复评审问题时：

1. 读取评审报告中的具体修复指令
2. **根据问题所在步骤定位修复起点**：
   - 问题在架构/模块划分（Step 2）→ 从 Step 2 开始，修改后需重新确认骨架
   - 问题在数据模型（Step 3）→ 从 Step 3 定点修复
   - 问题在接口设计（Step 4）→ 从 Step 4 定点修复
   - 问题在功能模块流程（Step 5）→ 从 Step 5 定点修复
   - 问题在非功能/三板斧（Step 6-7）→ 从对应 Step 定点修复
3. **定点修复** — 只改评审指出的章节，不扩散到其他步骤
4. 更新需求追踪矩阵（如有变化）
5. 重新执行 Step 9 方案检查
