# 系分编写执行指南

> designer 子Agent 在 Phase 3 使用。目标：生成结构化系分文档 + 需求追踪矩阵。

## Input

| 文件 | 必须 | 说明 |
|------|------|------|
| `{需求目录}/需求清单.md` | 是 | 需求条目列表，每条需求必须有对应设计 |
| `{需求目录}/代码阅读报告.md` | 是 | 现有代码结构、约束、风险 |
| `{需求目录}/知识摘要.md` | 是 | 技术约束和业务规则（设计决策依据） |
| EXTENSION_DOCS.design | 否 | 业务知识文档、数据字典等 |
| DESIGN_TEMPLATE_URL | 否 | 外部系分模板（空则用内置模板） |

## Execution

### Step 1: 理解需求与上下文

1. 读取需求清单，逐条理解每条需求的描述和验收标准
2. 读取知识摘要，掌握技术约束和现有模块结构
3. 读取代码阅读报告，了解现有接口、数据模型、调用链
4. 如有 EXTENSION_DOCS.design，读取业务知识文档

### Step 2: 生成骨架并确认

1. 按设计文档模板（已注入prompt）生成系分文档骨架（仅章节标题）
2. 将骨架写入 `{需求目录}/系分文档.md`，然后**结束本轮执行，返回主Agent**
3. 主Agent 将骨架转交用户确认（designer 不直接与用户对话）
4. 用户确认后，主Agent Resume designer 继续 Step 3

### Step 3: 生成需求追踪矩阵

> 需求追踪矩阵.md 是**独立文件**，不是系分文档的一部分。

逐条需求填写映射：

```markdown
# 需求追踪矩阵

| 需求ID | 需求描述 | 设计章节 | 接口 | 数据变更 | 状态 |
|--------|---------|---------|------|---------|------|
| R01 | 手机号注册 | §4.1 | POST /api/v1/register | t_user 新增 | ✅ 已覆盖 |
| R02 | 唯一性校验 | §4.1 | 同上 | idx_phone UNIQUE | ✅ 已覆盖 |
```

**每条需求必须有映射**。如果某条需求无法映射到具体设计章节，标记为"❌ 未覆盖"并说明原因。

### Step 4: 填充系分文档内容

按骨架逐章填充，遵循以下规则：

1. **架构约束** — 严格遵循架构约束（已注入prompt）中的 MUST / MUST NOT
2. **技术约束** — 遵循知识摘要.md中提取的技术约束
3. **接口设计** — 按设计文档模板中的接口格式，每个接口标注关联需求ID
4. **数据模型** — 按设计文档模板中的数据变更格式
5. **时序图** — 每条核心流程用 Mermaid sequenceDiagram

### Step 5: 自检

在产出前自检：

- [ ] 需求追踪矩阵中**每条需求**都有对应的设计章节
- [ ] 每个接口都标注了关联需求ID、幂等机制、鉴权方案
- [ ] 每个数据变更都标注了容量预估和迁移方案
- [ ] MUST 约束 6 条逐条核对
- [ ] MUST NOT 约束 5 条逐条核对
- [ ] 所有引用标注来源

## Output

产出**两个独立文件**，写入 `{需求目录}/`：

| 文件 | 说明 |
|------|------|
| `系分文档.md` | 完整系分文档，格式见设计文档模板（已注入prompt） |
| `需求追踪矩阵.md` | 独立文件，需求→设计章节→接口→数据变更的映射表 |

### 文档质量要求

- [ ] 系分文档包含所有骨架章节，无空章节
- [ ] 每个接口有完整字段：Path、Auth、Idempotent、关联需求、Request、Response、Error Codes
- [ ] 每个数据变更有完整字段：变更类型、字段表、索引、迁移、容量
- [ ] 至少一张核心流程时序图
- [ ] 需求追踪矩阵覆盖率 = 100%（所有需求都有映射）

### 日志

写入 `{需求目录}/log/phase3-系分编写.md`

### 修正模式（Resume）

当主Agent Resume 本Agent修复评审问题时：
1. 读取评审报告中的具体修复指令
2. **定点修复** — 只改评审指出的章节
3. 更新需求追踪矩阵（如有变化）
4. 重新执行 Step 5 自检
